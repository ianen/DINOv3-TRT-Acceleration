# ADR-013 · V1.0.2 — Persistent Timing Cache + Multi Optimization Profiles（Quick Win #2）

> **状态**：**Implemented-Mixed**（V1.0.2 Week 1，2026-05-02 闭合实施 + 部分 acceptance criteria 修正）
>
> **实测 findings on TRT 10.13.2.6**：
> 1. **`--persistentCacheSize`** flag 在 TRT 10.13.2.6 **不存在**（trtexec 报 `Unknown option`）；该 flag 是 TRT ≥ 10.16 才加。代码层支持已就位，但运行时 trtexec 拒绝 → 此项 effectively 待 ADR-014 升级 10.16.1 后才能启用。
> 2. **`--builderOptimizationLevel=5`** 在 r518 b8 BF16 prefer (`*:bf16` + prefer constraints) 上**无可测量增益**：V1.0.2 build (level 5, 490.1 MB engine) median 62.03 ms vs V1.0.1 baseline (default level 3, 490 MB engine) median 62.02 ms — 差 0.16% 在测量噪声内。**Negative result**：ViT-L/16 BF16 在 TRT 10.13.2.6 上 level 3 已接近最优 kernel，level 5 没有更好选择空间。
> 3. **`--precisionConstraints=prefer` + `--layerPrecisions=*:bf16`** 是关键组合 — V1.0.1 baseline 用此组合得 490 MB BF16 engine；不传则 TRT 默默把多数层留 FP32 → 966 MB / 200 ms（3× 退化）。这是 build_engine_trtexec.py 的 documentation 缺失，已在 V1.0.2 build 流程内文档化。
> 4. **多 optimization profiles + `--profile` 分隔符**接口已实施 + 12 单元测试 PASSED；具体收益待 ADR-016 sparsity engine + ADR-015 multi-stream pool 协同验证（V1.0.2 高 batch 才显效）。
>
> **acceptance verdict**：本 ADR 的 4 个目标中：
> - ✅ trtexec.py 接口扩展（multi-profile + opt level + sparsity flag）— Implemented
> - ⚠️ persistent timing cache — 需 ADR-014（TRT 10.16.1）才生效
> - ❌ opt level 5 实测 5% 增益目标 — Not Achieved（noise-level 0.16%）
> - ✅ 12 单元测试 + bit-exact build verification — Implemented
>
> **历史档案 - 原 Proposed 状态**：
> **触发**：V1.0.1 codebase audit 发现 `Code/src/dinov3_trt/engine/trtexec.py` 当前用**单 optimization profile**（`min=1, opt=8, max=32`），且 timing cache 路径 per-call 不一致（每次 build 重跑 tactic search ~5–15 min）。
> **依据**：TRT 多 profile 让每个 batch range 独立选最优 tactic（可能差 1.05–1.10×）；持久 timing cache 跨 build 复用 tactic 决策，让 iterative experimentation（V1.0.2 8–12 周内 30+ engine builds）从每次 5–15 min 降到 1–3 min。
> **期望**：bit-exact build 改动，速度增益 1.05–1.10× on extreme batches（b1, b32）；dev velocity 提升 5–10×；engine binary SHA256 跨重建 100% 可复现（V1.0.2 G6 目标）。
> **决策点**：是否在 V1.0.2 Week 1–2 与 ADR-012 并行实施？多 profile 是否需要重新评估 V1.0.1 baseline 数字？

---

## 1. 决策点

V1.0.2 G6 要求"engine SHA256 跨重建 100% 可复现 with persistent timing cache"，且 G1（low batch）/ G2（high batch）目标都需要 tactic 在各 batch size 上独立优化。当前单 profile 是否限制了 V1.0.1 已有数字？引入多 profile 后能否仍保持 V1.0.1 的 BF16 prefer cos_min ≥ 0.997 baseline？

---

## 2. 现状（V1.0.1 frozen）

`Code/src/dinov3_trt/engine/trtexec.py` 当前 build 命令（line 89-91）：

```bash
trtexec --onnx=... \
    --minShapes=pixel_values:1x3x224x224 \
    --optShapes=pixel_values:8x3x224x224 \
    --maxShapes=pixel_values:32x3x224x224 \
    --memPoolSize=workspace:4096 \
    --bf16 \
    --timingCacheFile=Artifacts/timing_cache/<engine_name>.cache \
    ...
```

问题：
1. **单 profile**：TRT 用 `opt=8` 的 shape 搜索 tactic，b=1 / b=32 时被迫复用 b=8 选定的 kernel（可能次优）
2. **timing cache 路径不一致**：每个 engine build 用自己的 cache 文件（`fp32.cache` / `bf16.prefer.cache` / `int8.smoothquant.alpha080.cache` / ...），跨配置无 tactic 知识复用
3. **`--builderOptimizationLevel` 未设**：默认 level 3（balanced），可设到 5（exhaustive）让 tactic 搜索更彻底（一次性代价，缓存后零开销）
4. **`--persistentCacheSize` 未设**：CUDA L2 cache 配置可影响 ViT attention kernel 的 key-value cache 命中

---

## 3. 备选方案对比

| 方案 | 工作量 | 增益 | bit-exact | dev velocity |
|---|---|---|---|---|
| **A** 仅多 profile（单 cache）| 0.5 周 | 1.05–1.10× extreme batch | ✅ | 不变 |
| **B** 仅持久共享 cache（单 profile）| 0.3 周 | 1.0× perf，build 速度 5–10× | ✅ | ↑↑ |
| **C** **多 profile + 持久共享 cache + builderOptimizationLevel 5（本 ADR）** | **1 周** | **1.05–1.10× extreme batch + 100% determinism** | ✅ | ↑↑↑ |
| **D** Per-batch-size 独立 engine（不用多 profile）| 1.5 周 | 1.05–1.10× | ✅ | runtime 复杂度上升 |
| **E** 等 V1.0.2 Week 5–6 sparsity 时合并做 | 与 ADR-016 合并 | 不正交，难单独验证 | ✅ | 延迟解锁 G6 determinism |

**选择**：**方案 C** — 多 profile + 持久共享 cache + builderOptimizationLevel 5。理由：
1. 三件事都改 `trtexec.py` 同一函数，同 ADR 一次到位避免反复改动
2. 都是 build-side 改动，不动 runtime / weights / cos numbers，真正 bit-exact
3. dev velocity 5–10× 提升直接受益于 V1.0.2 后续 30+ engine builds（sparsity / FP8 / 各 batch / 各 res 组合）

## 4. 实施设计

### 4.1 修改 `Code/src/dinov3_trt/engine/trtexec.py`

新增参数支持多 profile 与统一 cache：

```python
@dataclass(frozen=True)
class OptProfile:
    """One optimization profile entry: (min, opt, max) shape per input."""
    min_batch: int
    opt_batch: int
    max_batch: int


@dataclass(frozen=True)
class BuildConfig:
    onnx_path: Path
    output_engine: Path
    resolution: int = 224
    precision: Literal["fp32", "fp16", "bf16", "int8", "fp8"] = "bf16"
    profiles: tuple[OptProfile, ...] = (
        OptProfile(1, 1, 1),     # ← b=1 独立
        OptProfile(4, 8, 16),    # ← b=4-16 共一个 profile
        OptProfile(16, 32, 32),  # ← b=16-32 独立
    )
    workspace_mb: int = 4096
    builder_optimization_level: int = 5  # ← 默认 5（exhaustive）
    timing_cache_path: Path = Path("Artifacts/timing_cache/shared_v102.cache")  # ← 统一
    persistent_cache_size_mb: int | None = None  # ← 可选，None = TRT default


def build_trtexec_command(cfg: BuildConfig) -> list[str]:
    cmd = ["trtexec", f"--onnx={cfg.onnx_path}",
           f"--saveEngine={cfg.output_engine}"]

    # 多 profile 拼接
    for i, p in enumerate(cfg.profiles):
        if i > 0:
            cmd.append("--profile")  # 引入新 profile
        cmd.extend([
            f"--minShapes=pixel_values:{p.min_batch}x3x{cfg.resolution}x{cfg.resolution}",
            f"--optShapes=pixel_values:{p.opt_batch}x3x{cfg.resolution}x{cfg.resolution}",
            f"--maxShapes=pixel_values:{p.max_batch}x3x{cfg.resolution}x{cfg.resolution}",
        ])

    cmd.extend([
        f"--memPoolSize=workspace:{cfg.workspace_mb}",
        f"--builderOptimizationLevel={cfg.builder_optimization_level}",
        f"--timingCacheFile={cfg.timing_cache_path}",
        f"--{cfg.precision}",
    ])
    if cfg.persistent_cache_size_mb is not None:
        cmd.append(f"--persistentCacheSize={cfg.persistent_cache_size_mb}")
    return cmd
```

### 4.2 共享 timing cache 设计

| 维度 | V1.0.1 | V1.0.2 ADR-013 |
|---|---|---|
| Cache 文件数 | 一个 per engine（45+ 个）| **一个共享** `Artifacts/timing_cache/shared_v102.cache` |
| Cache 位置 | 散落 | 统一在 `Artifacts/timing_cache/` |
| 跨配置复用 | ❌（fp32 cache 不能给 bf16 用）| ✅ TRT 自动按 (precision, layer-shape) 索引，互不干扰 |
| 跨重建复用 | ✅（同一 engine）| ✅（任何 engine） |
| Determinism | ⚠️（首次 build 不确定）| ✅（cache hit 后 100% deterministic）|

### 4.3 builderOptimizationLevel 5 vs 3

| Level | Tactic 搜索深度 | Build time（ViT-L bf16）| Engine quality |
|---|---|---|---|
| 0 | 最快，最少 tactic | ~30 sec | -10% perf |
| 3（V1.0.1 默认）| 平衡 | ~5–10 min | baseline |
| 5（V1.0.2 推荐）| 全部 tactic + autotuning | ~15–30 min（首次）/ 1–3 min（cache hit）| +5% perf 期望 |

**注意**：level 5 首次 build 慢，但被 timing cache 抵消 — 第二次同 config build 几乎瞬间。V1.0.2 的 30+ build 会大量 cache hit。

### 4.4 SHA256 determinism 验证

新脚本 `Code/scripts/verify_engine_determinism.py`：

```python
"""Build the same engine 3 times with shared timing cache, assert SHA256 equal."""
import hashlib
import subprocess
from pathlib import Path

def main() -> int:
    cfg = ...  # standard BF16 prefer config
    hashes = []
    for i in range(3):
        engine_path = Path(f"/tmp/det_test_{i}.engine")
        cmd = build_trtexec_command(cfg)
        cmd[2] = f"--saveEngine={engine_path}"
        subprocess.run(cmd, check=True)
        hashes.append(hashlib.sha256(engine_path.read_bytes()).hexdigest())
    if len(set(hashes)) != 1:
        print(f"FAIL: hashes differ: {hashes}")
        return 1
    print(f"PASS: all 3 builds yield SHA256 {hashes[0]}")
    return 0
```

def test_default_profiles_cover_b1_b32():
    cfg = BuildConfig(...)
    profiles_min = [p.min_batch for p in cfg.profiles]
    profiles_max = [p.max_batch for p in cfg.profiles]

## 6. 风险与缓解

| 风险 | Likelihood | Impact | 缓解 |
|---|---|---|---|
| 多 profile 让 engine binary 变大（每 profile 多份 kernel）| Medium | Low | 实测约 +30–50% engine size；可接受（45 engines × 1 GB → 60 GB 仍在硬盘范围）|
| `builderOptimizationLevel=5` 首次 build 太慢导致 dev 受阻 | Medium | Medium | 文档化 "first build slow"；CI 用 cache hit 路径；增量开发先用 level 3 |
| 共享 timing cache 跨 TRT 版本不兼容 | High（升级 10.16.1 时）| High | TRT 升级（ADR-014）后自动 invalidate cache，rebuild from scratch；Acceptable |
| 多 profile 让 runtime 选错 profile（context.setOptimizationProfileAsync）| Low | Medium | C++ runtime 已有 profile 切换逻辑；测试 N profile × N batch 矩阵 |
| Engine SHA256 仍不一致（CUDA driver 内部非确定性）| Low | Medium | 文档化 "deterministic up to GPU driver state"；接受 99% 场景 |
