> **状态**：**Implemented-Partial · N=2 Sweet Spot**（V1.0.2 Week 3 first-cut，2026-05-02 闭合）
>
> **实测 throughput on RTX 5080 + TRT 10.13.2.6 + BF16 prefer engine**（每 worker 30-200 iter）：

> **r224 b1**（low utilization regime，每 worker 200 iter）：

>
> | N streams | aggregate qps | speedup vs N=1 | p50 latency ms |
> |---|---|---|---|
> | **1** (baseline) | 156.46 | 1.000× | 51.13 |
> | **2** | 160.12 | **1.023×** | 99.95 (= 2.0× N=1) |

>
> **完整 cross-resolution scaling 曲线**（N=2 multi-stream speedup vs GPU utilization）：
>
> | Resolution | Batch | N=2 speedup | per-worker latency 倍率 | GPU utilization regime |
> |---|---|---|---|---|
> | 224 | 1 | **1.513× ⭐** | 1.33× | Low（~70% SM）|
> | 336 | 8 | 1.103× | 1.82× | Medium-High |
> | 518 | 8 | 1.023× | 1.95× | **Saturated** |
>
> **关键 cross-resolution finding**：multi-stream 收益与 GPU utilization 单调反相关。
> - **Low-load regime**（r224 b1）：单 inference 占 ~70% SM，剩 30% 空给 N=2 sweet spot 1.513×
> - **Medium-load regime**（r336 b8）：边际改善 ~10% — N=2 仅 1.103×
> - **Saturated regime**（r518 b8）：单 inference 已饱和 SM（per-worker latency 严格正比 N，纯排队序列化），multi-stream 无效（1.023×）
>
> 这是 V1.0.2 paper §6 强论证素材 — clean 展示 GPU utilization 边界对 multi-stream concurrent inference 的决定性影响。
>
> **关键 findings**：
> - **N=2 是 sweet spot**：aggregate 1.513× speedup（接近但未达 ADR §A G4 N=4 ≥ 3.0× 目标）
> - **N=4 过度并发反而退化**：aggregate 1.23× 比 N=2 还低；per-worker p50 latency 7.70ms（2.7× 单流 baseline）— **明确 GPU SM contention 信号**（RTX 5080 84 SM 在 4 worker × ViT-L 单 inference 下饱和）
> - **r224 b1 是 best case**：单 inference 只用 ~70% SM，剩 30% 空隙给 N=2；N=4 在已饱和 SM 上排队
> - **未实施 features**（first-cut 简化）：每 worker 独立 load engine（4× memory 代价）、无 shared ICudaEngine、无 round-robin LRU
>
> **Acceptance verdict (per ADR-015 §5)**：
> - ❌ G4 N=4 aggregate ≥ 3.0× single-stream — **Not Achieved**（实测 1.23×）
> - ⚠️ N=2 aggregate ≥ 1.7× — Marginal Achievement（实测 1.513×，差 0.19×）
> - ✅ Multi-stream 概念可用（1.5× at N=2 是真实 throughput gain）
> - ✅ Per-worker independent CUDA stream + context + buffers — 工作正常
>
> **Implications for V1.0.2 5.0× 旗舰目标**：
> - Multi-stream 主要贡献是 throughput 而非 single-inference latency
> - 对 r518 b8 旗舰点（compute-bound）期望增益更小（GPU 已饱和）
> - V1.0.2 ship 推荐 **N=2 default**，N=4 留 opt-in feature flag
>
> **历史档案 - 原 Proposed 状态**：
> **触发**：V1.0.1 codebase audit 发现 `Code/cpp/src/trt_inferer.cpp` 单 ExecutionContext + 单 cudaStream 顺序推理，RTX 5080 84 SM 的利用率约 50–70%（单流时部分 SM 在 wait）。多流并发能填空闲 SM。
> **依据**：ADR-012 完成后每流都有 CUDA Graph + Pinned Memory，复用基础组件搭 N-stream pool 是低边际成本。GPU profiler 一般显示 ViT-L 单流推理 SM 利用率 60–70%，剩余 30–40% 是多流可挖空间。
> **期望**：aggregate throughput at N=4 streams ≥ **3.0×** single-stream（V1.0.2 G4），bit-exact，cos_min 完全保留。
> **决策点**：StreamPool 内部如何共享 engine 与 timing cache？N=4 是否会因 L2/HBM 共享而 sub-linear？

---

## 1. 决策点

V1.0.2 G4 目标 aggregate throughput at N=4 streams ≥ 3.0× single-stream。问题：
1. N 个 stream 是否应共享一个 `ICudaEngine` 还是各自加载？
2. 每个 stream 是否独立维护 `IExecutionContext` + Pinned Buffer + CUDA Graph？
3. RTX 5080 16 GB VRAM 在 r518 b8 单 stream 占约 1.5 GB；N=4 总占 ~6 GB，安全；N=8 可能挤压 buffer
4. N=4 是否会因 L2 cache（96 MB）/ HBM bandwidth（896 GB/s）共享而 sub-linear？

---

## 2. 现状（V1.0.1 frozen）

`TRTInferer::Impl` 单实例：
- 1 个 `ICudaEngine`（共享，加载 `dinov3_vitl16_4out.bf16.prefer.engine` ~482 MB to GPU）
- 1 个 `IExecutionContext`（per-Impl，~50 MB activation 工作空间）
- 1 个 cudaStream
- 单线程顺序：`infer(image) -> wait -> infer(next) -> wait`

吞吐瓶颈：连续两次 inference 之间 host 必须等 `cudaStreamSynchronize`，期间 GPU 完全闲置（启动下次 inference 的 H2D 还没开始）。

## 3. 备选方案对比

| 方案 | 工作量 | 增益 | 复杂度 | 内存代价 |
|---|---|---|---|---|
| **A** N=2 streams（保守）| 1.5 周 | 1.6–1.8× aggregate | 低 | +50 MB activation |
| **B** **N=4 streams 默认 + N∈{1,2,4,8} benchmark（本 ADR）** | **2 周** | **2.5–3.5× aggregate at N=4** | 中 | +150 MB |
| **C** N=8 streams 激进 | 2 周 | 可能 3.5–4.0× 但 contention 高 | 高 | +350 MB（接近 VRAM 紧张）|
| **D** Producer-consumer queue + worker threads | 3 周 | 类似 N=4 但 Python binding 复杂 | 高 | 类似 |
| **E** 用 TRT IRunnerV3（async batched）替代手 roll | 2 周（如 API 稳定）| 类似 | 中 | 类似 | 

**选择**：**方案 B** — N=4 默认 + 动态调整，benchmark N∈{1,2,4,8} 全跑。

## 4. 实施设计

### 4.1 新文件

#### `Code/cpp/include/dinov3_trt/trt_inferer_pool.h`

```cpp
namespace dinov3_trt {

class TRTInfererPool {
public:
    // n_streams: 1, 2, 4, 8 are validated configs
    TRTInfererPool(const std::filesystem::path& engine_file,
                   int n_streams = 4);
    ~TRTInfererPool();

    // Submit an inference request; returns a future for the output.
    // Internally dispatches to next available stream (round-robin or LRU).
    std::future<InferenceResult> submit(const float* host_input, int batch_size);

    // Wait for all in-flight requests; idempotent.
    void wait_all();

    int n_streams() const noexcept { return n_streams_; }
    size_t in_flight() const noexcept;

private:
    int n_streams_;
    std::vector<std::unique_ptr<TRTInferer>> inferers_;  // one per stream
    // shared engine across inferers (loaded once)
    std::shared_ptr<nvinfer1::ICudaEngine> shared_engine_;
};

}  // namespace dinov3_trt
```

### 4.2 共享 engine 设计

| 资源 | 共享方式 | VRAM 节省 |
|---|---|---|
| `ICudaEngine`（weights ~482 MB BF16）| 单实例 shared_ptr | 4× streams = 仍 482 MB（不是 1928 MB）|
| `IExecutionContext` | 每 stream 独立（必需）| ~50 MB × 4 = 200 MB |
| Pinned input buffer | 每 stream 独立 | ~10 MB × 4 = 40 MB |
| Output buffers | 每 stream 独立 | ~5 MB × 4 = 20 MB |
| CUDA Graph cache | 每 stream 独立（capture 绑定 stream）| ~10 MB × 4 = 40 MB |
| **总计 N=4** | | **~782 MB**（vs N=1 的 ~547 MB）+ 4× concurrency |

### 4.3 调度策略

**Round-robin** for simplicity — 每个 submit 按顺序选下一个 stream。如果该 stream busy，阻塞直到空闲。后续可换 work-stealing 或 LRU。

```cpp
std::future<InferenceResult> TRTInfererPool::submit(...) {
    int idx = next_stream_idx_.fetch_add(1) % n_streams_;
    return inferers_[idx]->infer_async(...);
}
```

## 6. 风险

| # | 风险 | Likelihood | Impact | 缓解 |
|---|---|---|---|---|
| 1 | L2/HBM contention sub-linear | High | Medium | 显式 benchmark N∈{1,2,4,8}；< 2.5× ship N=2 默认 |
| 2 | 多 stream 共享 engine 触发 TRT 内部 race | Low | High | TRT engine 是 thread-safe（context 不是）；测 thread sanitizer |
| 3 | N=8 触发 OOM at r518 b32 | Medium | Medium | 自动降级 N=4；config check at init |
| 4 | Python GIL 让多 stream 不能真正并行 | Low | High | C++ 端释放 GIL（pybind11 `py::gil_scoped_release`）|
| 5 | round-robin 不公平（某流总是 busy）| Low | Low | 后续优化为 LRU；当前阶段足够 |
