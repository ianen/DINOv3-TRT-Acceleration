# ADR-016 · V1.0.2 — 2:4 Structured Sparsity（NVIDIA ASP）

> **状态**：**Proposed**（V1.0.2 Week 5–6，未实施 — **首个有 cos 风险的 V1.0.2 项**）
> **触发**：V1.0.1 G3 memory-bound 边界（r336 b16+ / r518 b16）显示 FP32 与 BF16 都被 weight bandwidth 主导（speedup ~1.0×）。2:4 structured sparsity 让 50% weight 为零，硬件天然跳过零项 → weight bandwidth 减半 → 直接破 BW wall。
> **依据**：Ampere（sm_80）以来 NVIDIA Tensor Core 原生支持 2:4 sparse matmul，Blackwell sm_120 进一步优化；ASP（Automatic SParsity）库提供 PyTorch-side mask 生成；TRT ≥ 10.16 提供 production-grade sparse kernel selection。
> **期望**：1.2–1.5× speedup on weight-bound 层（attention QKV proj + MLP），cos_min ≥ 0.99（V1.0.2 G5 严格 gate），cos_min 可能掉 0.005–0.020；如不耐受则 ship 部分层 sparsity 或降为 stretch result。
> **决策点**：哪些层应用 sparsity？精度耐受怎么测？V1.0.1 BF16 prefer 0.9977 在加 sparsity 后能否仍 ≥ 0.99？

---

## 1. 决策点

V1.0.2 G3 要求 r518 b16 cpp end-to-end ≥ 1.6×（V1.0.1 ~1.0× BW saturated）。其他 V1.0.2 quick wins（CUDA Graphs / Pinned / Multi-Stream）不破 BW wall — 只有 sparsity 真正减少 weight 传输。但 sparsity 是首个**改 weights** 的 V1.0.2 项，cos 风险不可忽略。

---

## 2. 现状（V1.0.1 frozen）

V1.0.1 不用 sparsity：
- ONNX 导出后 weights 全密集（dense）
- TRT engine 用 dense Tensor Core kernel
- BW saturation at r336 b16+ / r518 b16+：FP32 与 BF16 latency 几乎相同（~1.0× speedup），证明瓶颈是 memory subsystem 不是 compute

`Code/src/dinov3_trt/quantization/` 不含 sparsity 模块；`Code/src/dinov3_trt/engine/trtexec.py` 不传 `--sparsity=enable` flag。

---

## 3. 2:4 sparsity 机制

**Pattern**：每 4 个连续 weight 中至少 2 个为 0（mask = `[1,1,0,0]` 或类似 6 种排列）。

| 维度 | Dense | 2:4 Sparse |
|---|---|---|
| Weight 占用 | 100% | **50%**（其他半为零，不存储）|
| Memory bandwidth | baseline | **0.5×**（halved）|
| Compute throughput | 1× | **2×**（sparse Tensor Core）|
| 对 BW-bound 层 | baseline | **2× peak**（实际 1.2–1.5×） |
| 对 compute-bound 层 | baseline | **2× peak**（实际 1.3–1.7×） |
| 精度损失 | 0 | 0.5–2% （depends on layer sensitivity）|

Per-layer 适用性：
- ✅ **attention QKV projection / output projection**（大 matmul，BW-bound 重灾区）
- ✅ **MLP up/down projection**（最大的 matmul）
- ⚠️ **patch embedding conv**（kernel 小，sparsity ROI 低）
- ❌ **LayerNorm / softmax**（无 weight，不适用）

---

## 4. 备选方案对比

| 方案 | 工作量 | speedup | cos 风险 | 难点 |
|---|---|---|---|---|
| **A** ASP magnitude-based mask（粗粒度，全 attention+MLP）| 1 周 | 1.2–1.4× weight-bound | cos_min 可能掉 0.020 | 一次性 mask 不重训 |
| **B** **Per-layer ablation + selective sparsity（本 ADR）** | **2 周** | **1.2–1.5×** | **cos_min 风险可控（仅耐受层）** | 需 ablation 矩阵 |
| **C** Sparsity + 部分 fine-tuning（< QAT 但 > pure mask）| 3 周 | 1.3–1.5× | cos_min 损失 ~0.005 | 训练资源 + 数据 |
| **D** 等 V1.3 QAT 时同时做 sparsity + QAT | 与 ADR-011 合并（≥ 1 月）| 1.4–1.6× | 最低（可恢复）| 训练强依赖 |
| **E** 不做 sparsity，仅靠 V1.0.2 其他向量 | 0 | 仍受 BW wall 限 | 0 | G3 目标 ≥ 1.6× 不可达 |

**选择**：**方案 B** — Per-layer ablation 后 selective sparsity，retrain-free。理由：
1. V1.0.2 周期内 retrain 不可行（V1.3 QAT 才做训练）
2. ablation 能精准识别耐受层，避开破坏 cos 的层
3. 即使最差情况只有 ~1/3 层耐受，也能拿到 ~1.15× boost

"""Generate 2:4 structured sparsity masks via NVIDIA ASP-style algorithm."""
from __future__ import annotations
import numpy as np
from pathlib import Path

def compute_2to4_mask(weight: np.ndarray) -> np.ndarray:
    """Compute 2:4 mask: per group of 4 along last dim, keep top-2 by magnitude.
    
    Returns boolean mask same shape as weight.
    """
    *batch, last = weight.shape
    assert last % 4 == 0, f"last dim must be divisible by 4, got {last}"
    grouped = weight.reshape(*batch, last // 4, 4)
    abs_w = np.abs(grouped)
    # top-2 indices per group of 4
    top2_idx = np.argsort(-abs_w, axis=-1)[..., :2]  # shape (..., 2)
    mask = np.zeros_like(grouped, dtype=bool)
    np.put_along_axis(mask, top2_idx, True, axis=-1)
    return mask.reshape(*batch, last)

## 7. 风险

| # | 风险 | Likelihood | Impact | 缓解 |
|---|---|---|---|---|
| 1 | 全部层 sparsity 让 cos_min < 0.99 | High | High | per-layer ablation 优先；选择性应用 |
| 2 | TRT 10.13.2.6 sparsity kernel fallback dense（无效）| High | High | ADR-014 升级 10.16.1 是前置依赖 |
| 3 | ASP magnitude-based mask 不是 optimal | Medium | Medium | V1.0.3 / V1.3 QAT 可能改 fine-tuned mask；本 ADR 接受 ASP baseline |
| 4 | r518 b16 sparsity 仍 BW-bound（未达 G3）| Medium | Medium | 多 batch 验证；不达标降级为 stretch |
| 5 | sparsity engine binary 大小变化导致 deployment 复杂 | Low | Low | engine size 不变（masked weight 仍占位）|
