# ADR-019 · V1.0.3 Triton Inference Server Integration

- **Status**: Proposed → **Parked (user-blocker: Windows host has neither Docker Desktop nor WSL2 distro)**
- **Date**: 2026-05-02
- **Related**: V1.0.3 plan §4.1, V1.0.3 plan §8 Risk #1 (predicted), ADR-014 (TRT 10.16.1 upgrade also user-blocked), ADR-020 (C++ pool also blocked)

## 1. Context

V1.0.3 plan §4.1 designates Triton Inference Server as the **A1 主推** (primary recommended) path to throughput-oriented serving:

- Industry-standard reusable solution (NVIDIA's official model server)
- Direct TensorRT engine support (no model rewrite)
- Built-in dynamic batching + concurrent model instances
- "Almost a 70% increase in throughput" per NVIDIA's own multi-instance benchmark numbers
- `perf_analyzer` provides Pareto-frontier latency-throughput sweeps out of the box

The plan §8 **Risk #1** explicitly anticipated:
> Triton Server Windows binary 不稳 / 需 Linux — Likelihood High, Mitigation: Docker WSL2 fallback；或承接 cloud Linux 实例

This risk has now **materialized**.

## 2. Probe (2026-05-02)

Probe of the Windows RTX 5080 host (`ssh windows-pc 'cmd /c "docker --version & wsl --status & wsl --list --verbose"'`):

| Dependency | Status |
|---|---|
| `docker` CLI in PATH | ❌ "不是内部或外部命令" (not installed) |
| WSL default version | WSL 1 (downgrade from default WSL 2) |
| WSL distros installed | ❌ none ("用于 Linux 的 Windows 子系统没有已安装的发行版") |
| nvidia-container-toolkit | N/A (no Docker host to install into) |

**Conclusion**: Triton has no native Windows binary. The standard distribution path is the NGC Triton Docker container (`nvcr.io/nvidia/tritonserver:25.x-py3`). With neither Docker Desktop nor a WSL2 distro available, Triton **cannot be installed without user-side action** on the host.

## 3. Three Unblock Paths

Listed in order of recommended preference for the V1.0.3 timeline:

### Path A — Install Docker Desktop on Windows (recommended)

- Effort: ~1-2 hr (download ~3 GB + reboot + WSL2 backend enable)
- Cost: $0 (Docker Desktop is free for educational/non-commercial use)
- Requires: Windows admin password
- Outcome: full V1.0.3 plan §4.1 ADR-019 path opens, including dynamic batching grid search (ADR-021)
- Caveat: Docker Desktop on Windows uses WSL2 + nvidia-container-toolkit; needs `wsl --install` first to provision Ubuntu 22.04 distro

### Path B — Cloud Linux GPU instance (fallback)

- Effort: ~30 min to spin up
- Cost: ~$0.7-1.2 USD/hr (AWS g5.2xlarge L4, GCP a2-highgpu-1g A100)
- Caveat: must rsync engine + benchmark scripts; loses the RTX 5080 sm_120 specificity (cloud has L4/A100/H100 — different SM count, different cache)
- Outcome: opens Triton path but on a different GPU; results would be analytical reference, not RTX 5080 final numbers

### Path C — Skip Triton entirely

- Effort: 0
- Cost: 0
- Outcome: V1.0.3 ships without G6 (Triton vs custom pool comparison) and without ADR-021 (dynamic batching grid). G1/G2/G3 throughput targets stay attached to ADR-020 C++ pool path only — and ADR-020 is itself blocked by TRT 10.13 Myelin thread-safety (see ADR-020 §9.2)
- Documented as V1.0.3 known-limitations; ADR-019 marked "Proposed-Parked" not "Implemented-Negative"

## 4. Decision (provisional, awaiting user input)

**Default path = C (skip)** until user actively chooses A or B. Rationale:

- C is zero-friction: V1.0.3 still ships meaningful deliverables (G7 utilization 4 regime acceptance, ADR-020 Phase 1, plan + ADR documentation, M5/M6 paper section "throughput limits" written from the comprehensive G7 saturation data we collected today)
- A and B cost real wallclock + cognitive load; only worth paying if user wants the Triton numbers in the report

If user picks **A**, work plan:

1. User: install Docker Desktop on Windows + `wsl --install` + reboot
2. Agent: validate via `ssh windows-pc 'docker run --rm hello-world'` and `docker run --rm --gpus all nvcr.io/nvidia/k8s/cuda-sample:nbody nbody -gpu`
3. Agent: pull `nvcr.io/nvidia/tritonserver:25.04-py3` (~6 GB), set up `Code/triton/dinov3/` model repository with `config.pbtxt` (instance_group + dynamic_batching), launch via `docker compose up -d`
4. Agent: run `perf_analyzer` grid search (ADR-021) → CSV → figures
5. Agent: G6 cross-tool compare vs ADR-020 C++ pool (if/when C++ pool also unblocks)

If user picks **B**, work plan:

1. User: provision AWS g5.2xlarge or GCP a2-highgpu-1g, share SSH access
2. Agent: rsync `Code/Artifacts/engines/dinov3_vitl16_4out.bf16.prefer.engine` + scripts, run Triton + perf_analyzer, mark all numbers as "cloud reference, not RTX 5080"
