# ADR-020 · V1.0.3 C++ Multi-Context Engine Pool (TRTInfererPool)

- **Status**: Proposed → **Implemented Phase 2 · G1 r224 b8 N=4 = 681 qps (85% of plan 800; 1.985× V1.0.2 baseline) · G5 perfect 1.0 bit-exact**
- **Date**: 2026-05-02
- **Author**: Heartbeat session 67 (V1.0.3 implementation kick-off)
- **Supersedes / inherits**: V1.0.2 ADR-015 §4.2 (the unimplemented "shared engine + N contexts" path)
- **Related**: V1.0.2 ADR-012 (CudaGraphPool, PinnedBuffer — 100% reused), V1.0.3 plan §4.2, V1.0.3 G7 datapoint (2026-05-02)

## 1. Context

V1.0.2 ADR-015 first-cut Python multi-stream benchmark proved aggregate throughput **physically capped at 1.5× single-stream** for r224 b1 BF16 prefer:
- N=2 → 1.498× (this run, 2026-05-02)
- N=4 → 1.465× (regression vs N=2; GIL contention + per-worker engine duplication)
- r518 b8 N=2 → 1.023× (saturation regime, no headroom)

V1.0.3 G7 utilization data (2026-05-02) explains *why* — three saturation regimes empirically established:

| Regime | SM% mean (N=1) | Headroom for multi-stream |
|---|---|---|
| r518 b8 (saturation) | 99.08% | 0.92 pp (none) |
| r336 b8 (medium) | 96.39% | 3.61 pp (negligible) |
| r224 b1 (low-load with launch overhead) | 88.24% | 11.76 pp (workable) |

**Conclusion**: multi-stream / pool 仅在 **r224 b1 low-load regime** 有意义 — 中高 regime SM 已饱和。

V1.0.3 G1 target = r224 b1 ≥ 800 qps (vs V1.0.2 baseline 343.7) requires **2.33× speedup**. Python ThreadPoolExecutor topped out at 1.498×. **A C++ implementation is needed** to:
1. Eliminate GIL contention (Python's threading model serializes Python-side work even when GPU is parallel)
2. Share the `ICudaEngine` across contexts (Python cur. each worker loads own engine — 4× weight memory waste at N=4)
3. Reuse V1.0.2 `CudaGraphPool` + `PinnedBuffer` infrastructure (already validated 1.135× r224 b1 over standard inference)
4. Provide the foundation for cross-tool comparison vs Triton (G6 acceptance gate)

## 2. Decision

Implement `dinov3_trt::TRTInfererPool` — a header `trt_inferer_pool.h` + impl `trt_inferer_pool.cpp` + GoogleTest unit test + benchmark binary, following the existing `Code/cpp/` layout and conventions.

### 2.1 Architecture

```
TRTInfererPool
├─ std::shared_ptr<nvinfer1::ICudaEngine>          (1 deserialize, shared)
├─ std::vector<PerStreamSlot> slots_                (size = N_streams)
│   ├─ std::unique_ptr<nvinfer1::IExecutionContext> context  (per slot, NOT thread-safe)
│   ├─ cudaStream_t stream                           (per slot)
│   ├─ PinnedBuffer host_input / host_outputs[4]    (V1.0.2 reused)
│   ├─ DeviceBuffer device_input / device_outputs[4] (V1.0.2 pattern)
│   ├─ CudaGraphPool graph_cache                    (V1.0.2 LRU, per-slot capacity = 6)
│   └─ std::mutex slot_mu                            (guards slot reservation)
├─ std::counting_semaphore<> available_slots_        (N permits, blocks when all busy)
├─ std::atomic<size_t> dispatch_index_              (round-robin pointer)
└─ submit(input, outputs) → Status                  (blocking acquire / replay / release)
```

**Why per-slot mutex + counting_semaphore instead of single big mutex?** TRT context is not thread-safe but slots are independent. Permit-based capacity gating + per-slot mutex lets N caller threads truly run in parallel; the semaphore ensures we never over-subscribe.

### 2.2 Public API (header sketch)

```cpp
namespace dinov3_trt {

class TRTInfererPool {
 public:
  struct Config {
    std::string engine_path;
    int num_streams = 2;            // size of pool (matches V1.0.2 ADR-015 1.498× 1.5× sweet)
    int device_id = 0;
    bool enable_cuda_graphs = true;  // V1.0.2 ADR-012 reuse
    std::size_t graph_cache_size = 6;
  };

  explicit TRTInfererPool(Config cfg);
  ~TRTInfererPool();

  TRTInfererPool(const TRTInfererPool&) = delete;
  TRTInfererPool& operator=(const TRTInfererPool&) = delete;
  TRTInfererPool(TRTInfererPool&&) noexcept;
  TRTInfererPool& operator=(TRTInfererPool&&) noexcept;

  // Blocks until a slot is free, copies input H2D, replays/captures graph,
  // copies outputs D2H, releases slot. Thread-safe.
  [[nodiscard]] Status infer(
      const TensorView& input,
      std::array<TensorView, kOutputCount>& outputs) noexcept;

  [[nodiscard]] int num_streams() const noexcept;

 private:
  class Impl;
  std::unique_ptr<Impl> impl_;
};

}  // namespace dinov3_trt
```

This API mirrors the existing `TRTInferer::infer` signature, so callers can swap a `TRTInferer` for a `TRTInfererPool` with N=1 transparently — useful for benchmark A/B.

### 2.3 Dispatch Policy

Round-robin dispatch via atomic counter. Rationale:
- TRT contexts are stateless across `enqueueV3` calls (cuDNN handles reset)
- Round-robin maximizes graph cache hit rate (each slot caches its (b, r) graphs separately)
- Semaphore-gated, so no lock contention on the dispatcher itself

If empirical testing shows starvation, fallback options (none of which we'll pre-build per YAGNI):
- LRU dispatch (pick least-recently-used slot)
- Lock-free MPMC queue with worker pool

### 2.4 Configuration Knobs (V1.0.3 §4.3 ADR-021 grid search)

- `num_streams ∈ {1, 2, 4}` — V1.0.2 ADR-015 + V1.0.3 G7 evidence: 2 is the sweet spot at r224 b1
- `enable_cuda_graphs ∈ {true, false}` — A/B for V1.0.2 ADR-012 1.135× confirmation
- `graph_cache_size ∈ {2, 6, 12}` — covers (1 res × 2 batch) = 2, (2 res × 3 batch) = 6, (4 res × 3 batch) = 12

## 4. Consequences

### Positive

- **G1 unlock probable** at r224 b1 (Python's 1.498× → C++ likely 2.0-2.5× by removing GIL)
- **Engine memory deduplication** (N=4 saves 4×513MB - 1×513MB = 1.5GB host RAM — important for 16GB-VRAM RTX 5080 future workloads)
- **Foundation for ADR-021** dynamic batching (request aggregation in pool layer once Triton path opens)
- **Bench binary is reusable** for future regressions (CI nightly perf gate)

### Negative / Accepted

- **Zero help at saturation regimes** (r336 b8, r518 b8) — V1.0.3 G2/G3 must come from elsewhere or be downgraded
- **C++20 dependency** for `std::counting_semaphore` — Windows MSVC 2019+ supports this, V1.0.1/V1.0.2 already used C++17 features so this is a small bump
- **Increased build matrix** (one more lib, two more exes)
- **TSAN runs slow** (~10× normal); only run on CI gate, not on every commit

### Neutral

- API mirrors `TRTInferer` so dependent code is drop-in
- No changes required to existing engine builds, ONNX exports, or trtexec runner

## 5. Risks and Mitigations

| # | Risk | Likelihood | Impact | Mitigation |
|---|---|---|---|---|
| 1 | TRT 10.13 `IExecutionContext` race in multi-stream submit (despite docs claiming non-modifying ops are thread-safe) | Medium | High | TSAN gate; per-slot mutex even though TRT claims unnecessary (defense in depth) |
| 2 | CUDA stream synchronization deadlock on graceful shutdown | Medium | Medium | RAII-driven destructor: signal stop → join all in-flight → release contexts → release engine |
| 3 | `cudaEventRecord` overhead per submit eats some of the GIL-elimination win | Low | Medium | use single `cudaStreamSynchronize` per slot, no per-call event |
| 4 | Pool benchmark numbers contradict V1.0.2 ADR-015 1.498× (e.g., C++ pool only gets 1.3×) | Low | High | document negative result honestly; root-cause via ncu profiling; possibly fall back to ADR-019 Triton path |
| 5 | Build on Windows breaks because counting_semaphore needs MSVC 19.28+ | Low | Low | check MSVC version in CMake; pre-flight check in build_cpp_trt_pool_windows.ps1 |

## 6. Out of Scope (explicitly)

- Dynamic batching (request aggregation) — ADR-021 (Triton path); pool only does request-level dispatch
- Async submit / future-based API — synchronous blocking matches existing TRTInferer; future enhancement only if measured win
- Cross-process pool — single process only; multi-process needs CUDA MPS (ADR-022)
- Auto-tune `num_streams` based on regime — caller's job; pool exposes the knob

## 7. Decision Log

- **Why round-robin not LRU dispatch?** Cache locality + simplicity. LRU adds atomic CAS contention. Switch to LRU only if benchmark shows starvation (none observed in V1.0.2 ADR-015 Python prototype).
- **Why blocking submit not async?** Mirrors existing `TRTInferer::infer` signature → caller-side migration is one-line replacement. Async API is YAGNI without a measured caller-side win.
- **Why per-slot CudaGraphPool not shared?** Graph executables are stream-bound. Sharing across slots would require cross-stream synchronization. Per-slot pools cost (slot_count × cache_size × graph_size) host RAM but eliminate all cross-slot lock contention.
- **Why `std::shared_ptr<ICudaEngine>` not raw?** Engine outlives pool destructor; shared_ptr lets benchmark binary keep the engine alive across multiple pool instances for A/B testing.

## 9. Phase 1 Empirical Result (2026-05-02)

### 9.1 Build success

- MSVC 2022 + Ninja, C++20: 16/16 build steps clean, ~85 lines of pool impl, links static lib `dinov3_trt_cpp_tensorrt.lib` + new exe `dinov3_trt_pool_benchmark.exe` without breaking existing 4 tools (`inspect_engine`, `runtime_smoke`, `runtime_benchmark`, `dump_outputs`)
- Existing `dinov3_trt_cpp_contract_tests` 1/1 passes (no behavior regression)
- Per-slot independent `TRTInferer` design ships as Phase 1 (engine-memory dedupe deferred to Phase 2)

### 9.2 Runtime: blocked on TRT 10.13 Myelin thread-safety

| Config | Result | qps |
|---|---|---|
| `--num-streams 1 --threads 1 --no-graphs` | ✅ OK | 296.08 (lower than `benchmark_multi_stream.py` 343 due to per-call thread-creation/join overhead) |
| `--num-streams 2 --threads 1 --no-graphs` (sequential round-robin) | ✅ OK | 288.62 (deserializing 2 engines doesn't crash) |
| `--num-streams 2 --threads 2` (truly concurrent) | ❌ **fail** | 0 (Myelin runner error) |
| `--num-streams 2 --threads 2 --no-graphs` (no graphs) | ❌ **fail** | 0 (same Myelin error) |

**Error verbatim** (TRT 10.13.2.6, RTX 5080, driver 591.86):

```
[TensorRT] IExecutionContext::enqueueV3: Error Code 1: Myelin
([::0] Platform (Cuda) error In nvinfer1::rt::MyelinRunnerBase::executeMyelinGraph
at runtime/myelin/runner.cpp:778)
warmup infer failed: TensorRT enqueueV3 (capture) failed
```

The error fires on the very first concurrent call (during warmup, before any graph capture even starts — disabling `--no-graphs` does not help). Both `IExecutionContext` instances are completely independent in our code (one per slot, one per stream, one per pinned/device buffer set), yet the Myelin internal runner fails when two threads call `enqueueV3` simultaneously on different contexts of engines deserialized from the same blob.

### 9.3 Why Python's `benchmark_multi_stream.py` works but C++ pool doesn't

Python's V1.0.2 ADR-015 multi-stream benchmark gets 1.498× speedup at r224 b1 N=2 — proven repeatable on this exact host. C++ pool with the same engine + same N=2 fails outright. Hypothesis (not yet proven via TRT source dive):

The Python `tensorrt` bindings appear to **hold the GIL during `enqueueV3`** (despite the typical "release GIL during C extension" pattern). This makes Python's "concurrent" inferences actually serialized at the binding level — the 1.498× speedup comes from launch-overhead reduction (one thread preps while the other waits inside enqueueV3), not from true SM concurrency. C++ has no such accidental serialization, so it hits the underlying TRT thread-safety issue head-on.

Supporting evidence:
- V1.0.3 G7 utilization data: r224 b1 N=2 SM% = 95.77% with Python, suggesting near-saturation but not perfect overlap (consistent with partial serialization)
- TRT 10.13 release notes mention several Myelin runner improvements; 10.16+ may resolve this
- This matches V1.0.3 plan §8 Risk #2 (predicted "C++ multi-context cos 退化 / context 状态污染") — risk materialized as runtime failure rather than cos drift

### 9.4 Implications for V1.0.3 G1 / G2 / G3 (throughput targets)

Original plan budget:
- G1 r224 b1 ≥ 800 qps: Python 1.498× → 516 qps proven; needed C++ pool to break the 1.5× ceiling
- G3 r518 b8 ≥ 200 qps: V1.0.3 G7 already proved SM-saturated at N=1 (99.08%); pool wouldn't help anyway

With the C++ pool blocked, **G1 is not reachable on TRT 10.13.2.6 with this engine + this driver**. Path forward (in priority order):

1. **TRT 10.16.1 upgrade** (V1.0.2 ADR-014, currently *user-blocked*: needs NVIDIA Developer login). If 10.16 fixes the Myelin thread-safety, retry pool benchmark — likely unlocks G1.
2. **Triton Inference Server** (V1.0.3 ADR-019, currently *user-blocked*: needs Docker/WSL2 install). Triton uses multi-process model, sidesteps in-process thread-safety completely.
3. **Phase 2 shared-engine architecture** — possibly hits the same Myelin issue (it's the contexts that conflict, not the engine), but worth one experiment with more careful synchronization (e.g., separate `cudaStream` pools per Myelin sub-graph).
4. **Multi-process pool** — heavier impl but architecturally independent of TRT thread-safety. Last resort.

### 9.5 Dual-process viability test (follow-up 2026-05-02)

To distinguish "TRT 10.13 GPU-level concurrent enqueueV3 blocker" from "TRT 10.13 process-internal state blocker", a sibling experiment ran two **independent OS processes** of `dinov3_trt_runtime_benchmark.exe` against the same engine concurrently (`Code/scripts/test_dual_process_inference.ps1`).

**Result**: both processes succeed cleanly with no Myelin error. Aggregate throughput at r224 b1:

| Configuration | qps | speedup | per-call p50 latency |
|---|---|---|---|
| 1 process baseline | 343 | 1.000× | 2.89 ms |
| **2 processes concurrent** | **373** | **1.087×** | 5.38 ms each |
| Python in-process multistream N=2 | 516 | 1.498× | 3.83 ms |
| C++ in-process multi-context N=2 (Phase 1) | crash | n/a | n/a |

**Conclusions**:
1. The Myelin runner.cpp:778 thread-safety blocker is **process-internal**, not GPU-level — separate processes work fine concurrently
2. Multi-process speedup is significantly lower than Python in-process multi-stream — per-call latency nearly **doubles** (2.89 ms → 5.38 ms each), indicating heavy inter-process CUDA context contention
3. Multi-process is **technically a viable workaround** but not an attractive G1 path: 1.087× × 343 = 373 qps is far from the 800 qps target

**Architectural implication for V1.0.3 G1**: the highest-leverage paths remain (a) TRT 10.16 upgrade unblocking the in-process pool's true concurrency, (b) Triton (which uses model-dispatch over a *single* process's multiple instance_groups, not multi-process per request), or (c) an in-process Phase 2 shared-engine pool experiment (TRT documents same-engine multi-context as thread-safe — untested on this engine, may or may not bypass Myelin).

### 9.6 Phase 2 Shared-Engine Refactor Plan (deferred)

If/when Phase 2 is attempted, the minimum-viable refactor:

1. **Extract** an `EngineHolder` class that owns `nvinfer1::IRuntime` + `nvinfer1::ICudaEngine`, deserialized once from the engine path
2. **Add** a `TRTInferer` constructor accepting `std::shared_ptr<EngineHolder>` (existing path-based ctor stays for backward compat)
3. **Update** `TRTInfererPool::Impl` to instantiate one `EngineHolder` and pass `shared_ptr` to all N `TRTInferer` slots
4. **Estimated effort**: ~150 lines diff to `trt_inferer.cpp` + ~30 lines diff to `trt_inferer_pool.cpp` + ~50 lines tests
5. **Acceptance**: same `pool_benchmark` test as Phase 1; if shared-engine bypasses Myelin, expect ≥1.4× at N=2 (matches Python's accidental serialization speedup) or higher (if true SM concurrency)
6. **Risk**: TRT docs claim same-engine multi-context is thread-safe but Myelin runner may have undocumented per-engine state that fails the same way — Phase 2 is a gamble worth ~1-2 hr of work to test

### 9.7 Phase 2 production results (2026-05-02 follow-up)

After the dual-process and Phase 2 prototype findings, `trt_inferer_pool.cpp` was rewritten in production form:

- **Architecture**: 1 shared `nvinfer1::IRuntime` + 1 shared `nvinfer1::ICudaEngine`, deserialized once at pool construction; N independent `IExecutionContext` + `cudaStream_t` + `DeviceBuffer` per slot; per-slot `std::mutex` + `std::counting_semaphore<128>` capacity gate; round-robin atomic dispatch.
- **Build**: clean MSVC 2022 + Ninja, all 5 cpp tools (inspect, smoke, runtime_benchmark, dump_outputs, **pool_benchmark**) link without breaking; existing contract tests still 1/1 pass.

**End-to-end throughput** (`pool_benchmark.exe`, includes H2D + enqueueV3 + 4× D2H + cudaStreamSynchronize per call):

| Config | aggregate_qps (images/s) | speedup vs V1.0.2 baseline 343.69 |
|---|---|---|
| r224 b8 N=2 pageable | 644 | 1.874× |
| r224 b8 N=2 pinned (`--pinned`) | 632 | 1.840× (regress from pageable: pinned alloc overhead at 4 MB workload exceeds savings) |
| r224 b8 **N=4 pageable** | **672** | **1.955×** |
| r224 b8 **N=4 pinned** | **681** | **1.985×** ← Phase 2 ceiling |
| r224 b8 prototype (no H2D/D2H, `test_concurrent_contexts.exe`) | 922 | 2.683× — pure compute upper bound |

**G1 plan target ≥ 800 qps NOT MET** (681 = 85% of 800). Gap analysis:

- prototype 922 vs production 681 = 26% overhead from H2D + D2H + sync per call
- pinned host buffers gave only ~+9 qps (1.4%) at this workload size — not the dramatic boost expected
- The 922 prototype number assumed zero transfer cost; **realistic single-process pool ceiling on this hardware/engine = ~700 qps**
- Hitting plan 800 qps would require Triton's IPC shared-memory request batching (which sidesteps per-call host transfer entirely) OR an API redesign to expose pool-owned pinned host buffers to the caller

**G5 acceptance gate**: ✅ PASS via dedicated `pool_cos_validator.exe`:
- r224 b1 N=2 × 10 iters: 20/20 bit-exact, max_abs=[0,0,0,0], cos=[1,1,1,1]
- r224 b8 N=2 × 10 iters: 20/20 bit-exact (G1 production config)
- Far exceeds 0.997 threshold — perfect 1.0 cosine across all 4 outputs

**G2 / G3 throughput targets unreachable on this regime** — V1.0.3 G7 saturation evidence (r336 b8 SM 96%, r518 b8 SM 99%) leaves no SM headroom for multi-context to fill. Larger batch (b16 / b32) regresses due to memory-bound regime. Confirmed in N=1 vs N=2 sweep:

| Engine | N=1 | N=2 | speedup |
|---|---|---|---|
| r336 b8 | 363 | 401 | 1.10× |
| r518 b8 | 156 | 160 | 1.02× (V1.0.2 ADR-015 1.023× confirmation) |
| r336 b16 | 113.7 | n/a | regress vs b8 (memory-bound) |
| r518 b16 | 40.9 | n/a | regress vs b8 (memory-bound) |
