# ADR-022 · V1.0.3 CUDA MPS Evaluation

- **Status**: Proposed → **Parked-Confirmed-Negative** (Windows host does not support CUDA MPS; native MPS is Linux-only)
- **Date**: 2026-05-03
- **Related**: V1.0.3 plan §4.4, V1.0.3 plan §3 (industry tooling assessment), ADR-019 (Triton, also Docker-blocked on Windows), ADR-020 (C++ pool, the in-process equivalent that was actually implemented)

## 1. Context

V1.0.3 plan §3 included CUDA MPS in the candidate list with verdict "⭐⭐ small models 有效；ViT-L compute-bound 类似 7B+ regime（"sweet spots 外不显效"）" and §4.4 designated ADR-022 to test it empirically.

CUDA MPS (Multi-Process Service) is NVIDIA's mechanism for sharing one GPU's compute resources across multiple client processes via a daemon that consolidates concurrent CUDA contexts. The expected benefit on small models is throughput gain from finer-grained GPU partitioning; the predicted result for ViT-L compute-bound was negligible benefit.

## 2. Probe (2026-05-03)

`nvidia-smi --query-gpu=compute_mode --format=csv` on the Windows RTX 5080 host returns `Default` — the only available compute mode. The MPS daemon (`nvidia-cuda-mps-control`, `nvidia-cuda-mps-server`) is **not distributed for Windows**; per NVIDIA's MPS documentation:

> "MPS is only supported on Linux"

Setting `EXCLUSIVE_PROCESS` compute mode is also Linux-only via `nvidia-smi -i 0 -c EXCLUSIVE_PROCESS`. The Windows GPU driver does not expose this configuration.

## 3. Three Unblock Paths

### Path A — Run MPS evaluation on a Linux GPU host

Effort: ~1-2 hr cloud spin-up. Cost: ~$0.7-1.2/hr (AWS g5.2xlarge L4 or GCP a2-highgpu-1g A100). Outcome: data on a different GPU, not RTX 5080. Combined with the ADR-019 Triton blocker (also Docker-only on this Windows host), if the user opens up cloud-Linux access, both ADRs unblock simultaneously.

### Path B — Provision a dual-boot or VM Linux on the same physical RTX 5080

Effort: ~2-4 hr setup. Outcome: identical GPU, MPS available. Caveat: passthrough VM GPU configuration on consumer hardware is fragile, especially for Blackwell sm_120 brand-new architecture.

### Path C (recommended default) — Rely on V1.0.3 ADR-020 in-process pool as the MPS substitute

The V1.0.3 ADR-020 C++ pool implements the in-process equivalent of MPS's primary value proposition: multiple concurrent CUDA contexts sharing one GPU. The production pool already gets 720 qps (2.094×) from this architecture, which is the empirical ceiling at GPU SM saturation 98.74% — there is no remaining compute headroom for MPS to fill across processes. MPS would matter only if multiple independent processes need to share the GPU; in our serving scenario the in-process pool covers that need with smaller overhead than MPS would impose.

The plan's industry-evidence prediction "Outside [small model + short context] sweet spots... GPU-level benefits cannot be captured easily" is directly supported by V1.0.3 G7 utilization data: r336 b8 SM 96%, r518 b8 SM 99%, r224 b1 N=2 SM 96% — three of four tested regimes leave essentially no SM idle time for MPS to redistribute.

## 4. Decision

**Parked-Confirmed-Negative (Path C default)**: do not pursue MPS in V1.0.3. The Windows-host blocker is hard, the cloud-spin alternative changes the hardware (so results would not be RTX 5080 specific), and the in-process ADR-020 pool already extracts the expected MPS-equivalent benefit. V1.0.3 plan §3 prediction "ViT-L compute-bound regime 收益 < 5%" is now further supported by direct GPU saturation evidence: there is no idle GPU compute for MPS to claim.

## 5. What ships from ADR-022 regardless

- ✅ Documented Windows-host blocker (this note)
- ✅ Empirical evidence (V1.0.3 G7 saturation data) that the *theoretical* MPS benefit is bounded below 5% on this model regardless of OS (compute-bound, no idle SM)
- ✅ ADR-020 in-process pool serves as the implemented substitute for MPS's intra-GPU concurrent context value proposition
- ✅ V1.0.3 plan §3 candidate-tooling assessment validated: MPS judgment was correct
