# ADR-023 · V1.0.3 TensorRT-LLM and vLLM Inapplicability to ViT Pure Encoder

- **Status**: **Confirmed-Negative** (decision: do not pursue; document for paper §7 future-work reference)
- **Date**: 2026-05-02
- **Related**: V1.0.3 plan §3 (industry-tooling assessment), §4.5 (this ADR slot), V1.0.2 ADR-012 (CUDA Graph for ViT, already covered the only piece of vLLM that ViT could use)

## 1. Why this ADR exists

V1.0.3 plan §3 explicitly assesses TensorRT-LLM and vLLM as candidate frameworks for ViT throughput-oriented serving. The plan's preliminary verdict was "⭐ not applicable to pure encoder workloads." This ADR formalizes that verdict so:

1. The paper §7 future-work section can reference it instead of reasoning from scratch
2. Future readers / collaborators who ask "why didn't you use vLLM, it's so popular?" get a one-page authoritative answer
3. We reduce the risk of someone re-litigating this decision in a later iteration of the plan

## 2. The Frameworks

### 2.1 TensorRT-LLM

NVIDIA's open-source library for LLM inference acceleration on top of TensorRT. Core differentiators:

| Feature | What it optimizes | Applicable to ViT? |
|---|---|---|
| **In-flight batching (continuous batching)** | Concurrent generation of variable-length output sequences; allows new requests to join an active batch mid-decode | **No** — ViT has no decode loop; one forward pass produces all 4 feature outputs |
| **Paged KV cache** | Memory management for the attention key-value cache that grows with sequence length during autoregressive generation | **No** — ViT pure encoder has no KV cache to retain across decode steps |
| **Speculative decoding (Medusa, EAGLE)** | Predict multiple output tokens in parallel, validate with full model | **No** — no token generation in ViT |
| **Tensor parallel / pipeline parallel | Sharding very large LLMs across multiple GPUs | **No** — ViT-L/16 (~300M params) fits on one RTX 5080 with VRAM to spare; no need for sharding |
| **Custom attention kernels (FlashAttention v2/v3)** | Memory-bound attention optimization | **Partial** — V1.0.2 ADR-018 already evaluated custom attention kernel ROI on ViT and found it Limited-ROI (~7% gain). TRT-LLM's attention kernel could provide some benefit but is wrapped in LLM-specific orchestration. |

**Net**: TRT-LLM's value proposition is 90% LLM-decode-specific. Importing it into a ViT pipeline would require disabling or mocking the in-flight batcher, paged KV cache, and speculative decoding, leaving only the FlashAttention kernel — which V1.0.2 ADR-018 already documented as Limited-ROI on this model.

### 2.2 vLLM

UC Berkeley's open-source LLM inference engine. Core differentiators:

| Feature | What it optimizes | Applicable to ViT? |
|---|---|---|
| **PagedAttention** | KV cache memory paging (vLLM's flagship feature) | **No** — same reason as TRT-LLM |
| **Continuous batching** | Same concept as TRT-LLM in-flight batching | **No** |
| **Prefix caching** | Cache shared prefix tokens across requests for chat-style workloads | **No** — ViT processes independent images; no shared prefix |
| **Multi-lora serving** | Hot-swap LoRA adapters at request time | **No** — DINOv3 is a frozen backbone; no LoRA in our pipeline |
| **CUDA Graphs for ViT** (added in vLLM 0.5+) | Capture the forward pass as a CUDA graph for replay | **Yes — but already covered by V1.0.2 ADR-012** |

**Net**: vLLM's CUDA-Graph-for-ViT support is the single feature applicable to our workload, and V1.0.2 ADR-012 already implemented exactly this (with empirically validated 1.135× r224 b1 speedup, bit-exact). Importing vLLM solely for this feature would add a 2 GB Python dependency for code we already own.

## 3. What is genuinely shared across LLM serving and ViT serving

Despite the inapplicability of the LLM-specific features, three serving-tier capabilities **are** shared and **are** the subject of other V1.0.3 ADRs:

1. **Multi-context concurrent inference** (one engine, multiple execution contexts on multiple streams). This is the `IExecutionContext` pattern — V1.0.3 ADR-020 implements it directly in C++ for ViT, no LLM scaffolding needed.
2. **Dynamic request batching** (aggregate requests arriving in a small time window into one larger batch). This is V1.0.3 ADR-021 via Triton Inference Server's `dynamic_batching` config — Triton supports both LLM and non-LLM models uniformly; we use the non-LLM path.
3. **Multi-instance serving** (N copies of the model behind a load balancer). V1.0.3 ADR-019 via Triton's `instance_group` config — same Triton-server dispatcher serves both LLM and ViT models.

For ViT, these three capabilities are accessible *directly via Triton + plain TensorRT*; there is no value in routing through the LLM-specific serving stack.

## 4. Decision

**Do not adopt TensorRT-LLM or vLLM in V1.0.3.** Document this ADR for the paper, and proceed with:

- ADR-019 Triton (the LLM-agnostic serving layer) — currently parked on Docker
- ADR-020 C++ multi-context pool (in-process equivalent of Triton's instance_group) — currently shipped Phase 1, blocked at concurrent N=2 by TRT 10.13 Myelin
- ADR-021 dynamic batching tuning — depends on ADR-019

## 5. What If Someone Insists?

If a reviewer asks "why didn't you try vLLM with `--enforce-eager False` and see what happens?" — concrete answer:

- vLLM's `LLM` class requires a tokenizer (we have no text input)
- vLLM's image inputs go through `MultiModalRegistry` which expects a vision-language model paired backbone (LLaVA, Qwen-VL, etc.); standalone ViT-L/16 has no registered image-only inference path in vLLM 0.6.x
- vLLM 0.5+ added `--vision-language` flag that opens a path, but it still requires an LLM head; running just the vision encoder is not the supported entrypoint
- Estimated effort to get DINOv3 running through vLLM: ~3 days work for fork-and-patch + maintenance burden every vLLM release; expected outcome (per CUDA Graph for ViT analysis above): same ~1.135× speedup that V1.0.2 ADR-012 already achieves natively

The marginal cost-benefit fails any reasonable threshold.
