# ADR-026 · V1.0.4 Python production_benchmark.py — 6 段独立计时

- **Status**: **Code Implemented · Sweep Awaiting Windows SSH**（commit `3b0594b`，2026-05-07）
- **Date**: 2026-05-07
- **Related**: V1.0.4 plan §4.3 / §6 acceptance G3 + G4 + G5

## 1. Context

V1.0.0/1/2/3 已有 latency benchmark 脚本（`benchmark_multi_stream.py` 等），但都只测**单一总 latency**，不分阶段。V1.0.4 user 明确要求"**各部分耗时分别列出**" — 模拟生产环境推理流水线的**端到端时序分解**，6 段独立计时：

1. `disk_read` — 从磁盘读 jpg bytes
2. `jpg_decode` — JPEG decode → RGB ndarray
3. `preprocess` — float32/255 + ImageNet mean/std + HWC→NCHW
4. `h2d` — host → device memcpy + sync
5. `enqueueV3` — TRT context.execute_async_v3 + sync
6. `d2h` — 4 outputs device → host memcpy + sync

`resize` 不在 6 段内 — 数据集已离线 r=512（ADR-024）。

## 2. Decision

```bash
  --dataset Artifacts/datasets/good_r512 \

  --image-size 512 \
  --warmup 10 \
  --iters 100 \
  --output Artifacts/reports/v104_runs/r512_bf16_b8_py.json
```

### 2.2 计时方案

每段用 `time.perf_counter()` 独立采样：

```python
t0 = time.perf_counter(); raw = path.read_bytes()                  # disk_read
t1 = time.perf_counter(); img = PIL.open(io.BytesIO(raw)).convert("RGB")  # jpg_decode
t2 = time.perf_counter(); arr = preprocess(img)                    # preprocess
t3 = time.perf_counter(); cudart.cudaMemcpyAsync(...); sync()      # h2d
t4 = time.perf_counter(); context.execute_async_v3(...); sync()    # enqueueV3
t5 = time.perf_counter(); for o in outputs: D2H; sync()            # d2h
t6 = time.perf_counter()
timings.append(disk_read=t1-t0, jpg_decode=t2-t1, ...)
```

`cudaStreamSynchronize` 在每个 GPU 段后立刻调用 — 阻塞等待让计时反映**真实墙钟**而非 enqueue 延迟。

### 2.3 关键决策

| 决策 | 理由 |
|---|---|
| **JPEG decode 用 PIL** | 项目已有依赖，与 `evaluate_engine_pair_on_images.py` 一致；NumPy zero-copy |
| **Preprocess 用 NumPy（非 torch）** | 避免 torch 启动开销污染 timing；ImageNet mean/std 用 ndarray broadcast |
| **每段 sync 而非一次大 sync** | 6 段独立可观测；总 latency = 6 段之和（一致性可校验） |
| **Output 缓冲 NumPy float32** | BF16/FP16 输出时 numpy 缺 bf16 dtype，用 fp32 host buffer 接收（D2H 字节数按 engine dtype 算） |
| **Per-image trace 完整保留** | JSON 输出含每张图 6 段 trace，下游分析灵活（不只是 p50） |

### 2.4 输出 schema

```json
{
  "engine": "...", "dataset": "...",
  "batch_size": 8, "image_size": 512,
  "warmup": 10, "iters": 100,

}
```

## 4. 验收门

### G3 — 6 段独立计时

每个 (precision, batch) JSON 必须含 `stages_p50_ms.{disk_read, jpg_decode, preprocess, h2d, enqueueV3, d2h, total}` 7 个非零数字。一致性校验：
```
sum(stages_p50_ms[non-total]) ≈ stages_p50_ms.total（允许 ±2% 噪声）
```

### G4 — Pareto 数据点

8 必选数据点（4 batch × 2 必选精度）全部跑通 + JSON 落盘 → CSV 16 行（含 Python+C++）。

### G5 — Python ↔ C++ 对照

参考 ADR-027 — Python 与 C++ 同 (precision, batch) 6 段 p50 diff ≤ 10%。

## 5. Consequences

### Positive

- 生产环境分阶段诊断能力 — 直接看到 disk I/O / decode / preprocess / GPU 各占比，定位瓶颈不靠猜
- JSON 含 per-image trace — 后续可任意做 percentile / outlier 分析
- 与 V1.0.0/1/2/3 timing 模式同源（perf_counter + sync），方法学一致

### Negative / Trade-off

- 每段 sync 引入额外 host-side wait — 总 latency 比"一次 enqueue + 一次 sync"略大；但本研究目标是分阶段诊断，不是吞吐极限（吞吐看 V1.0.3 pool benchmark）
- Per-image trace 100 iters × 6 stages = 600 floats per JSON — 文件 ~50 KB / run，可接受

### Neutral

- 不嵌套调用其他 skill / framework，单文件可移植
