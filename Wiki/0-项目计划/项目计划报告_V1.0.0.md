## 0. 执行摘要

本项目针对 Meta DINOv3 视觉自监督基础模型（ViT-L/16，24 层），构建一条 **PyTorch → ONNX → TensorRT** 的完整推理加速管线，覆盖 **FP32 / FP16 / INT8** 三档精度，并提供 **Python 研究 API** 与 **C++ 生产推理封装** 双栈。

**与常规推理加速项目的关键差异**：DINOv3 推理输出取 transformer 第 **4 / 12 / 16 / 20** 层中间特征（多尺度密集预测特征），共 4 个 output binding。这一约束贯穿 ONNX 导出、引擎构建、INT8 校准、benchmark、精度对齐与跨语言封装的所有阶段。

**预期成果**：
- FP16 端到端加速 ≥ 1.8×（vs PyTorch eager），精度无损
- INT8 端到端加速 ≥ 2.5×，下游任务精度跌幅 ≤ 1%
- C++ 推理 wrapper 与 Python 数值一致（FP32 MaxAbsErr ≤ 1e-5）
- 完整 benchmark 矩阵：3 精度 × 4 batch × 3 分辨率
- 实验报告 + 可复现代码

## 1. 项目背景与目标

### 1.1 背景

DINOv3 是 Meta 发布的视觉基础模型，在分类、分割、深度估计等下游任务中表现优异，但其推理延迟在生产部署中是瓶颈。NVIDIA TensorRT 提供算子融合、精度量化、kernel 自动调优等加速能力。本项目研究 DINOv3 在 TRT 上的加速极限与精度边界，并形成可复用的工程实践。

### 1.2 目标（SMART）

| # | 目标 | 度量方式 |
|---|------|---------|
| G1 | 将 DINOv3 ViT-L/16 推理延迟降至 PyTorch baseline 的 ≤ 50%（FP16） | CUDA event p50/p95 测量 |
| G2 | INT8 量化加速 ≥ 2.5×，精度退化可量化、可控 | 4 个输出张量逐层 cosine sim ≥ 0.99；下游分类 top-1 跌幅 ≤ 1% |
| G3 | 同一 `.engine` 在 Python 与 C++ runtime 上数值一致 | 跨语言 parity 测试 MaxAbsErr ≤ 1e-5 (FP32) |
| G4 | 输出完整 benchmark 矩阵与技术报告 | 3 精度 × 4 batch × 3 分辨率 CSV + 可视化 |
| G5 | 代码可复现 | 单脚本一键复现，README 完整 |

### 1.3 范围（In / Out）

**In Scope**：
- ViT-L/16（主），ViT-H/14（如时间允许）
- 4 输出（layer 4/12/16/20）的多输出引擎
- FP32 / FP16 / INT8 三档精度
- ImageNet val 子集精度对齐
- Python + C++ 推理 runtime 封装
- 单 GPU 推理（不含 multi-GPU/分布式）

**Out of Scope**：
- 模型训练、微调、蒸馏
- 移动端/嵌入式部署（Jetson/边缘）
- 服务化部署（K8s / Triton Inference Server）
- 自定义 TRT plugin（仅在原生算子不支持时按需追加）

## 2. 关键技术约束

### 2.1 模型输出规约

DINOv3 推理需输出 transformer 第 **4 / 12 / 16 / 20** 层中间特征（含 CLS + patch tokens），对应 ViT-L/16（24 层）下：

| 输出 binding | 形状 | 用途 |
|--------------|------|------|
| `feat_layer_4` | `[B, 197, 1024]` | 浅层语义特征 |
| `feat_layer_12` | `[B, 197, 1024]` | 中层语义特征 |
| `feat_layer_16` | `[B, 197, 1024]` | 中高层语义特征 |
| `feat_layer_20` | `[B, 197, 1024]` | 高层语义特征 |

> 注：197 = 14×14 patch + 1 CLS token；1024 = ViT-L hidden size。该约束自动排除 ViT-S/B（仅 12 层）。

### 2.2 技术栈版本约束

| 维度 | 选型 |
|------|------|
| 模型权重 | DINOv3 ViT-L/16（facebookresearch/dinov3 官方） |
| 训练框架 | PyTorch ≥ 2.1 |
| 中间表示 | ONNX opset ≥ 17 |
| 推理引擎 | TensorRT ≥ 10.8（Blackwell sm_120 支持下限） |
| GPU 计算栈 | CUDA ≥ 12.8 + cuDNN 9.x（与主力机 RTX 5080 / Blackwell 匹配） |
| Python runtime | cuda-python（非 pycuda） |
| C++ binding | pybind11 |
| 校准器 | IInt8EntropyCalibrator2 |

## 3. 系统架构

### 3.1 分层架构图

```
┌─────────────────────────────────────────────────────────────────────┐
│                      Layer 5: User-Facing API                       │
│   Python: DinoV3TRTInferer  │  C++: TRTInferer  │  pybind11 Bridge │
└─────────────────────────────────────────────────────────────────────┘
                                  ▲
┌─────────────────────────────────────────────────────────────────────┐
│                  Layer 4: Runtime & Execution                       │
│  TRT IExecutionContext │ CUDA Stream Mgr │ Pinned Host Memory Pool │
│  Device Buffer (4 outputs × [B,197,1024]) │ Async H2D/D2H          │
└─────────────────────────────────────────────────────────────────────┘
                                  ▲
┌─────────────────────────────────────────────────────────────────────┐
│                Layer 3: Engine Build & Calibration                  │
│  TRT Builder │ Optimization Profile (min/opt/max) │ INT8 Calibrator│
│  Plan Cache (.engine binary, hash-keyed by model+precision+GPU)    │
└─────────────────────────────────────────────────────────────────────┘
                                  ▲
┌─────────────────────────────────────────────────────────────────────┐
│                  Layer 2: Model Conversion                          │
│   PyTorch DINOv3 ViT-L/16 → Forward Hook (layers 4/12/16/20)       │
│   → ONNX Exporter (opset 17, dynamic_axes) → onnx-simplifier       │
└─────────────────────────────────────────────────────────────────────┘
                                  ▲
┌─────────────────────────────────────────────────────────────────────┐
│              Layer 1: Source Model & Preprocessing                  │
│      Meta DINOv3 weights │ ImageNet-norm preprocessing pipeline     │
└─────────────────────────────────────────────────────────────────────┘
```

### 3.2 数据流

```
Image (HWC uint8)
  → Preprocess (resize+norm)  → Tensor [B, 3, 224, 224]
  → H2D async copy            → Device Input Buffer
  → TRT execute_async_v3      → 4× Device Output Buffers [B, 197, 1024]
  → D2H async copy + sync     → Dict[str, np.ndarray] (Python)
                              → std::array<Tensor, 4>   (C++)
```

## 4. 关键设计决策（ADR）

### ADR-001 · ONNX 多输出导出方式

| 项目 | 内容 |
|------|------|
| 决策点 | 从 ViT 抓取 4 个中间层输出 |
| **选择** | **Forward Hook + Wrapper Module** |
| 理由 | 不侵入第三方源；wrapper 显式 `return tuple` 让 ONNX 识别 4 个输出 |
| 权衡 | hook 需配合 `torch.jit.trace` 可见性；放弃零代码改动 |
| 影响 | `src/export/` 设计；输出 binding 命名固定 |

### ADR-002 · 动态 shape 策略

| 项目 | 内容 |
|------|------|
| 决策点 | 固定 vs 动态 |
| **选择** | **单 profile，仅 batch 维动态**（min=1, opt=8, max=32），分辨率固定 224×224 |
| 理由 | 分辨率动态会改变 token 数（197 是 14×14+1 产物），打破 4 输出形状契约 |
| 权衡 | 放弃多分辨率灵活性 |
| 影响 | `engine_builder` 接口；多分辨率需多 profile 重构 |

### ADR-003 · INT8 校准器选型

| 项目 | 内容 |
|------|------|
| 决策点 | Entropy / MinMax / Percentile |
| **选择** | **IInt8EntropyCalibrator2**（KL 散度），Percentile 99.99 作 A/B 对照 |
| 理由 | ViT 的 attention logits 与 GELU 激活长尾，MinMax 易被 outlier 拉伸；Entropy2 是 NVIDIA 官方推荐 |
| 权衡 | 校准时间长于 MinMax；放弃 per-channel 极致精度 |
| 影响 | `src/quantize/` 必须支持子集采样；保留 Percentile 路径 |

### ADR-004 · Python 推理 Runtime

| 项目 | 内容 |
|------|------|
| 决策点 | pycuda vs cuda-python |
| **选择** | **cuda-python**（NVIDIA 官方） |
| 理由 | pycuda 维护放缓；cuda-python 与 TRT 10.x / CUDA 12.x 深度对齐；与 C++ 行为一致更利跨语言对齐 |
| 权衡 | API 较底层、boilerplate 多 |
| 影响 | `src/infer/` 全部 CUDA 调用 |

### ADR-005 · C++/Python 接口边界

| 项目 | 内容 |
|------|------|
| 决策点 | pybind11 数据交换格式 |
| **选择** | **numpy buffer protocol 零拷贝 + 显式 ownership** |
| 理由 | 避免大 tensor 跨语言深拷贝；`py::array_t<float>` + `py::capsule` 管理生命周期 |
| 权衡 | 内存生命周期需文档化；放弃 std::vector 简单值传递 |
| 影响 | `cpp/bindings/` 须 GPU 调用期间释放 GIL |

### ADR-006 · 预/后处理位置

| 项目 | 内容 |
|------|------|
| 决策点 | CPU vs CUDA kernel vs TRT 内置 |
| **选择** | **预处理 CPU + 后处理 CPU**（V1.0.0） |
| 理由 | 优先保证 Python/C++ 数值位级一致；DINOv3 预处理仅 resize+归一化，非瓶颈 |
| 权衡 | 放弃 CUDA preprocessing 的 ~1–2ms 收益 |
| 影响 | `utils/preprocess.py` 与 C++ 等价实现共享同一组归一化常量；后续 V1.1 可迁至 NPP/CV-CUDA |

## 5. 关键接口契约

### 5.1 Python 推理 API

```python
class DinoV3TRTInferer:
    LAYER_NAMES = ("feat_layer_4", "feat_layer_12",
                   "feat_layer_16", "feat_layer_20")

    def __init__(self, engine_path: str, device_id: int = 0): ...

    def infer(self, img: np.ndarray) -> Dict[str, np.ndarray]:
        """
        Args:
            img: HWC uint8 [H,W,3] OR NCHW float32 [B,3,224,224]
        Returns:
            dict with 4 keys (LAYER_NAMES), each value shape [B,197,1024]
        Invariants:
            - keys 顺序稳定
            - 输出 dtype = np.float32（无论引擎精度）
            - 单次调用线程安全
        """

    def infer_batch(self, imgs: List[np.ndarray]) -> List[Dict[str, np.ndarray]]: ...
```

### 5.2 引擎构建器

```python
def build_engine(
    onnx_path: str,

    calibrator: Optional[IInt8Calibrator] = None,
    workspace_gb: int = 4,

    """Returns serialized engine bytes; precision='int8' requires calibrator."""
```

```cpp
class TRTInferer {
public:
    explicit TRTInferer(const std::string& engine_path, int device_id = 0);
```

## 6. 跨语言数值一致性策略

| 措施 | 实现 |
|------|------|
| **共享预处理常量** | `config/preprocess.yaml` 单一来源；Python/C++ 启动加载，禁止硬编码 mean/std |
| **位级一致性验证** | 单测：同一 PNG 经两端预处理后 `MaxAbsErr < 1e-7`；CI 强制 |
| **统一 stream 语义** | 两端均用非默认 stream，调用结束 `cudaStreamSynchronize` 后再读出 |
| **确定性数值** | TRT builder `kOBEY_PRECISION_CONSTRAINTS` + `kSTRICT_TYPES`；C++ 端 `CUBLAS_PEDANTIC_MATH` |
| **同一引擎二进制** | Python 与 C++ 共享同一份 `.engine`，仅 runtime 不同；engine SHA256 写入 benchmark 报告 |
| **对齐验证流程** | `tests/test_cross_lang_parity.py`：随机 10 张图 → Python 推理 → C++ 推理 → 4 输出逐元素对比，FP32 阈值 `< 1e-5`，FP16 `< 1e-2`，INT8 `cos_sim > 0.99` |
| **种子与随机性** | INT8 校准数据集顺序固定；预处理无随机 augmentation |

## 10. 风险管理

### 10.1 风险登记册（Top 8）

| # | 风险 | 概率 | 影响 | 缓解 | 应急方案 |

| **R1** | 4 层中间输出导致 ONNX 导出图结构异常（如 LayerNorm 折叠失败） | 中 | 高 | 提前 PoC 验证；`register_forward_hook` + 自定义 wrapper module | 退化为单输出 + 二次推理获取中间层 |
| **R2** | INT8 校准在 4 输出张量上精度崩塌 | 高 | 高 | 分层 sensitivity 分析；敏感层强制 FP16 混合精度；扩大校准集 | 放弃纯 INT8，采用 INT8/FP16 hybrid |
| **R3** | TRT 版本对 ViT 算子（GELU、LayerNorm）支持不全 | 中 | 中 | 锁定 TRT 10.x；预留 plugin 开发时间 | 自研 TensorRT plugin |
| **R4** | dynamic shape 引擎构建失败或性能不佳 | 中 | 中 | 使用 optimization profile，预设 min/opt/max | 退化为多个固定 shape 引擎 |
| **R5** | 显存不足以构建大 batch / 高分辨率 INT8 引擎 | 中 | 中 | 分批 build；workspace 调优；启用 timing cache | 降级 batch 上限，文档说明 |
| **R6** | C++ 与 Python 数值不一致（预处理差异） | 中 | 高 | 共享预处理常量；逐 stage 对齐；CI 强制 parity 测试 | 暂以 Python 版交付，C++ 列为 future work |
| **R7** | DINOv3 权重许可/下载受限影响复现 | 低 | 高 | 确认 license 并镜像权重 | 使用 DINOv2 ViT-L/14 作为兜底 |
| **R8** | 单/双人团队进度延期 | 中 | 中 | 周节奏跟踪；P6/P7 可裁剪 | C++ 封装裁剪为 minimal demo |

### 10.2 关键依赖链

- P2 依赖 P1：multi-layer hook 决定 ONNX 导出图结构
- P3 依赖 P2：4 输出 ONNX 是引擎构建输入
- P4 依赖 P3：FP16 通过后才做 INT8（共享构建脚手架与校准基础设施）
- P5 依赖 P3+P4：需所有精度引擎齐全
- P6 依赖 P5：选定最优引擎才封装 C++
- P7 横跨全程，但终稿依赖 P5/P6
- **关键路径**：P1 → P2 → P3 → **P4** → P5（INT8 是关键路径瓶颈）
