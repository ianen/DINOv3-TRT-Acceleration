## 0. 执行摘要

本项目针对 Meta DINOv3 视觉自监督基础模型（ViT-L/16，24 层），构建一条 **PyTorch → ONNX → TensorRT** 的完整推理加速管线，覆盖 **FP32 / FP16 / INT8** 三档精度，并提供 **Python 研究 API** 与 **C++ 生产推理封装** 双栈。

**与常规推理加速项目的关键差异**：DINOv3 推理输出取 transformer 第 **4 / 12 / 16 / 20** 层中间特征（多尺度密集预测特征），共 4 个 output binding。这一约束贯穿 ONNX 导出、引擎构建、INT8 校准、benchmark、精度对齐与跨语言封装的所有阶段。

> 📍 **V1.0.1 修订**：调研揭示 DINOv3 默认含 **4 个 register tokens**（HuggingFace `Dinov3Config.num_register_tokens=4` 硬编码）与 **RoPE 旋转位置编码**——这两点都不是 V1.0.0 的假设，相关修订见 §2.1、ADR-001、ADR-007。此外，**`IInt8EntropyCalibrator2` 自 TRT 10.1 起 deprecated**，主路径改为显式 Q/DQ 量化（NVIDIA Model Optimizer），见 ADR-003 修订。

**预期成果**（V1.0.1 调整）：

- FP16 端到端加速 **≥ 1.5× / stretch 1.8×**（vs PyTorch eager），精度无损
- INT8 端到端加速 **≥ 2.2× / stretch 2.5×**，下游任务精度跌幅 ≤ 1%
- C++ 推理 wrapper 与 Python 数值一致（**分档**：FP32 ≤ 1e-5、FP16 ≤ 1e-3、INT8 cos ≥ 0.99）
- 完整 benchmark 矩阵：3 精度 × 4 batch × 3 分辨率（含 p50 标准差 ≤ 5%、中位数去极值统计）
- 实验报告 + 可复现代码

## 1. 项目背景与目标

### 1.1 背景

DINOv3 是 Meta 发布的视觉基础模型，在分类、分割、深度估计等下游任务中表现优异，但其推理延迟在生产部署中是瓶颈。NVIDIA TensorRT 提供算子融合、精度量化、kernel 自动调优等加速能力。本项目研究 DINOv3 在 TRT 上的加速极限与精度边界，并形成可复用的工程实践。

### 1.2 目标（SMART）

> 📍 **V1.0.1 修订**：G1/G2 加速比阈值在三份调研一致建议下保守降调；G3 跨语言一致性改为分档表达。

| #  | 目标                                                                                                                  | 度量方式                                                                               |
| -- | --------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------- |
| G1 | 将 DINOv3 ViT-L/16 推理延迟降至 PyTorch baseline 的**≤ 67%（FP16，相当 1.5× 加速）；stretch ≤ 56%（1.8×）** | CUDA event p50/p95 测量，主力机 RTX 5080                                               |
| G2 | INT8 量化加速**≥ 2.2×（stretch 2.5×）**，精度退化可量化、可控                                                | 4 个输出张量**逐层逐输出独立**汇报 cosine sim ≥ 0.99；下游分类 top-1 跌幅 ≤ 1% |
| G3 | 同一 `.engine` 在 Python 与 C++ runtime 上数值一致                                                                  | **分档**：FP32 MaxAbsErr ≤ 1e-5；FP16 ≤ 1e-3；INT8 cos ≥ 0.99                 |
| G4 | 输出完整 benchmark 矩阵与技术报告                                                                                     | 3 精度 × 4 batch × 3 分辨率 CSV + 可视化；含 p50 std ≤ 5% 与中位数去极值            |
| G5 | 代码可复现                                                                                                            | 单脚本一键复现，README 完整，附 license 副本与 "Built with DINOv3" 标注                |

### 1.3 范围（In / Out）

**In Scope**：

- ViT-L/16（主），ViT-H/14（如时间允许）
- 4 输出（layer 4/12/16/20，即 0-based `[3,11,15,19]`）的多输出引擎
- FP32 / FP16 / INT8 三档精度
- ImageNet val 子集精度对齐
- Python + C++ 推理 runtime 封装
- 单 GPU 推理（不含 multi-GPU/分布式）

**Out of Scope**：

- 模型训练、微调、蒸馏
- 移动端/嵌入式部署（Jetson/边缘）
- 服务化部署（K8s / Triton Inference Server）
- 自定义 TRT plugin（仅在原生算子不支持时按需追加）

> 📍 **V1.0.1 新增 V1.1 Stretch Goals**（不计入本期范围，但为后续版本铺路）：FP8 PTQ via Model Optimizer（Blackwell 5th-gen Tensor Core）、SmoothQuant ViT-L PTQ、4 层组合 ablation（`[3,11,15,19]` vs `[5,11,17,23]` vs `[4,12,16,20]`）。

## 2. 关键技术约束

### 2.1 模型输出规约

> 📍 **V1.0.1 重大修正**：DINOv3 的实际 token 序列结构与 V1.0.0 假设不同。

DINOv3 ViT-L/16 LVD-1689M 默认含 **4 个 register tokens**（Darcet et al., 2023, "Vision Transformers Need Registers"）。完整 token 序列在 224×224 输入下为：

```
[CLS] + [reg₁ reg₂ reg₃ reg₄] + [196 patch tokens] = 201 tokens
```

**官方 `get_intermediate_layers()` API 默认裁剪 register tokens**，仅返回 `[CLS] + [196 patch] = 197 tokens`。

| 模式                                | Token 序列长度 | 形状               | 约束                                 |
| ----------------------------------- | -------------- | ------------------ | ------------------------------------ |
| **API 默认（裁剪 register）** | 197            | `[B, 197, 1024]` | 与 V1.0.0 一致；本项目主路径         |
| **保留 register tokens**      | 201            | `[B, 201, 1024]` | 仅当下游任务需要 register 信号时启用 |

> 📍 **V1.0.1 决策**：主路径沿用 197 tokens 形状（与 V1.0.0 计划兼容），但在 ADR-001 显式声明"裁剪 register tokens"为 architectural invariant，禁止隐式切换。

DINOv3 推理需输出 transformer 第 **4 / 12 / 16 / 20** 层（1-based；等价 0-based `[3,11,15,19]`）中间特征：

| 输出 binding      | 形状               | 用途           |
| ----------------- | ------------------ | -------------- |
| `feat_layer_4`  | `[B, 197, 1024]` | 浅层语义特征   |
| `feat_layer_12` | `[B, 197, 1024]` | 中层语义特征   |
| `feat_layer_16` | `[B, 197, 1024]` | 中高层语义特征 |
| `feat_layer_20` | `[B, 197, 1024]` | 高层语义特征   |

> 📍 **V1.0.1 来源说明**：4/12/16/20 这一组合**非 DINOv3 官方 canonical 推荐**，源自 DPT 风格密集预测任务的社区惯例（24 层 ViT-L 上常见 [5,11,17,23]，本项目选择偏中后层组合 [4,12,16,20]）。该约束自动排除 ViT-S/B（仅 12 层）。层组合的 ablation 已纳入 V1.1 stretch goal。

### 2.2 技术栈版本约束

> 📍 **V1.0.1 重大修订**：TRT 版本下限从 10.8 上调至 10.13（理想 10.16.1）；ONNX opset 从 17 上调至 18；PyTorch 改为稳定版优先。

| 维度           | 选型（V1.0.1）                                                                                                                             | 修订原因                                                                                    |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------ | ------------------------------------------------------------------------------------------- |
| 模型权重       | DINOv3 ViT-L/16 LVD-1689M（safetensors 格式）；HuggingFace `facebook/dinov3-vitl16-pretrain-lvd1689m`（需点击同意 license）              | 调研补充 license / 渠道 / 格式                                                              |
| 训练框架       | **PyTorch 稳定版 cu128 优先（2.7+ 已 OK）**；仅 exporter bug 触发时升 nightly 2.12.x                                                 | GPT 调研：稳定版已支持 cu128                                                                |
| 中间表示       | **ONNX opset ≥ 18**（RoPE 与 LayerNorm 原生算子需要）                                                                               | 调研：opset 17 RoPE 拆解为细粒度三角函数                                                    |
| 推理引擎       | **TensorRT ≥ 10.13（理想 10.16.1）**；10.8 仅作 baseline 对照                                                                       | TRT 10.10 已知 16% ViT 回归；10.14.1 修复 55% MHA Blackwell 回归；10.16.1 修复 78% FP8 回归 |
| GPU 计算栈     | CUDA ≥ 12.8 + cuDNN 9.x（与主力机 RTX 5080 / Blackwell 匹配）                                                                             | 不变                                                                                        |
| Python runtime | cuda-python（非 pycuda）                                                                                                                   | 不变                                                                                        |
| C++ binding    | pybind11                                                                                                                                   | 不变                                                                                        |
| 量化框架       | **NVIDIA TensorRT Model Optimizer（Q/DQ 显式量化）为主路径**；IInt8EntropyCalibrator2 已 deprecated（TRT 10.1+），仅作 baseline 对照 | GPT 调研独家发现                                                                            |
| 校准数据       | ImageNet-1k val 500–1000 张分层抽样，独立于评估集                                                                                         | 不变                                                                                        |
| License        | **DINOv3 License**（自定义协议，非 Apache/MIT），需附 license 副本与 "Built with DINOv3" 标注                                        | 调研补充                                                                                    |

## 3. 系统架构

### 3.1 分层架构图

```
┌─────────────────────────────────────────────────────────────────────┐
│                      Layer 5: User-Facing API                       │
│   Python: DinoV3TRTInferer  │  C++: TRTInferer  │  pybind11 Bridge │
└─────────────────────────────────────────────────────────────────────┘
                                  ▲
┌─────────────────────────────────────────────────────────────────────┐
│                  Layer 4: Runtime & Execution                       │
│  TRT IExecutionContext │ CUDA Stream Mgr │ Pinned Host Memory Pool │
│  Device Buffer (4 outputs × [B,197,1024]) │ Async H2D/D2H          │
└─────────────────────────────────────────────────────────────────────┘
                                  ▲
┌─────────────────────────────────────────────────────────────────────┐
│             Layer 3: Engine Build & Quantization                    │
│  TRT Builder (≥10.13) │ OptimizationProfile (静态分辨率, 动态batch) │
│  Q/DQ via ModelOpt PTQ │ Plan Cache (sha256: model+precision+arch+trt) │
└─────────────────────────────────────────────────────────────────────┘
                                  ▲
┌─────────────────────────────────────────────────────────────────────┐
│                  Layer 2: Model Conversion                          │
│  PyTorch DINOv3 ViT-L/16 → DinoV3Wrapper (RoPE source-edit)         │
│  → get_intermediate_layers([3,11,15,19]) → 4× [B,197,1024]          │
│  → ONNX Exporter (opset≥18, dynamic_axes={'B'}) → onnx-simplifier   │
└─────────────────────────────────────────────────────────────────────┘
                                  ▲
┌─────────────────────────────────────────────────────────────────────┐
│              Layer 1: Source Model & Preprocessing                  │
│  DINOv3 weights (safetensors, LVD-1689M license)                    │
│  ImageNet-norm preprocessing (mean/std centralized config)          │
└─────────────────────────────────────────────────────────────────────┘
```

> 📍 **V1.0.1 修订**：Layer 2 显式标注 RoPE 源码改造（避免 `If` 节点）；Layer 3 标注 Q/DQ 量化路径与版本下限。

### 3.2 数据流

```
Image (HWC uint8)
  → Preprocess (resize+norm)  → Tensor [B, 3, 224, 224]
  → H2D async copy            → Device Input Buffer
  → TRT execute_async_v3      → 4× Device Output Buffers [B, 197, 1024]
  → D2H async copy + sync     → Dict[str, np.ndarray] (Python)
                              → std::array<Tensor, 4>   (C++)
```

## 4. 关键设计决策（ADR）

### ADR-001 · ONNX 多输出导出方式 *（V1.0.1 修订）*

| 项目           | 内容                                                                                                                                                      |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 决策点         | 从 ViT 抓取 4 个中间层输出，含 register tokens 的处理策略                                                                                                 |
| **选择** | **DinoV3Wrapper Module + `get_intermediate_layers([3,11,15,19])` API**；register tokens 默认裁剪（API 默认行为），输出形状统一 `[B, 197, 1024]` |
| 理由           | API 默认裁剪 register 与 V1.0.0 既定形状契约一致；wrapper 显式 `return tuple` 让 ONNX 识别 4 个输出；不侵入第三方源（除 RoPE 必要改造，见 ADR-007）     |
| 权衡           | 放弃 register tokens 信号（部分下游任务可能受益）；如未来需要保留 register，需新建 ADR-001b                                                               |
| 影响           | `src/export/dinov3_wrapper.py` 设计；输出 binding 命名固定为 `feat_layer_{4,12,16,20}`                                                                |

### ADR-002 · 动态 shape 策略

| 项目           | 内容                                                                               |
| -------------- | ---------------------------------------------------------------------------------- |
| 决策点         | 固定 vs 动态                                                                       |
| **选择** | **单 profile，仅 batch 维动态**（min=1, opt=8, max=32），分辨率固定 224×224 |
| 理由           | 分辨率动态会改变 token 数（197 = 14×14+1+register 裁剪），打破 4 输出形状契约     |
| 权衡           | 放弃多分辨率灵活性                                                                 |
| 影响           | `engine_builder` 接口；多分辨率需多 profile 重构                                 |

### ADR-003 · INT8 量化方案 *（V1.0.1 重大修订）*

| 项目           | 内容                                                                                                                                                                                                        |
| -------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 决策点         | INT8 量化路径选型                                                                                                                                                                                           |
| **选择** | **主路径**：NVIDIA TensorRT Model Optimizer **显式 Q/DQ 量化**（Entropy 算法 + 99.99% Percentile A/B 对照）。**Baseline 路径**：legacy `IInt8EntropyCalibrator2`（仅作旧文献复现/对照） |
| 理由           | `IInt8EntropyCalibrator2` 自 TRT 10.1 起已 deprecated（GPT 调研独家发现）；ModelOpt 显式 Q/DQ 是 NVIDIA 官方当前推荐路径，支持 per-channel + per-tensor 混合、敏感层混合精度回退                          |
| 权衡           | ModelOpt 学习曲线略高；放弃 IInt8EntropyCalibrator2 的简洁性；但能获得更好的 ViT 精度（ViT attention logits 长尾对长期 deprecated API 不友好）                                                              |
| 影响           | `src/quantize/modelopt_ptq.py` 主实现；`legacy_calibrator.py` 仅对照；多输出**联合校准一次即可**（不需为 4 输出分别校准），但精度验收**必须逐输出汇报**                                     |

### ADR-004 · Python 推理 Runtime

| 项目           | 内容                                                          |
| -------------- | ------------------------------------------------------------- |
| 决策点         | pycuda vs cuda-python                                         |
| **选择** | **cuda-python**（NVIDIA 官方）                          |
| 理由           | pycuda 维护放缓；cuda-python 与 TRT 10.x / CUDA 12.x 深度对齐 |
| 权衡           | API 较底层、boilerplate 多                                    |
| 影响           | `src/infer/` 全部 CUDA 调用                                 |

### ADR-005 · C++/Python 接口边界

| 项目           | 内容                                                                              |
| -------------- | --------------------------------------------------------------------------------- |
| 决策点         | pybind11 数据交换格式                                                             |
| **选择** | **numpy buffer protocol 零拷贝 + 显式 ownership**                           |
| 理由           | 避免大 tensor 跨语言深拷贝；`py::array_t<float>` + `py::capsule` 管理生命周期 |
| 权衡           | 内存生命周期需文档化                                                              |
| 影响           | `cpp/bindings/` 须 GPU 调用期间释放 GIL                                         |

### ADR-006 · 预/后处理位置

| 项目           | 内容                                                                                      |
| -------------- | ----------------------------------------------------------------------------------------- |
| 决策点         | CPU vs CUDA kernel vs TRT 内置                                                            |
| **选择** | **预处理 CPU + 后处理 CPU**（V1.0.0/V1.0.1 阶段）                                   |
| 理由           | 优先保证 Python/C++ 数值位级一致；DINOv3 预处理仅 resize+归一化，非瓶颈                   |
| 权衡           | 放弃 CUDA preprocessing 的 ~1–2ms 收益                                                   |
| 影响           | `utils/preprocess.py` 与 C++ 等价实现共享同一组归一化常量；后续 V1.1 可迁至 NPP/CV-CUDA |

### ADR-007 · DINOv3 RoPE 节点处理 *（V1.0.1 新增）*

| 项目           | 内容                                                                                                                                                   |
| -------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------ |
| 决策点         | DINOv3 的 RoPE 含 `aten::if` 条件分支（DINOv2 不存在），ONNX 导出后保留为 `If` 节点，触发 TRT IIfConditionalOutputLayer 失败（Issue #4603, #4558） |
| **选择** | **首选源码改造**（直接修改 DINOv3 fork 中 RoPE 实现，分支静态化为 eval 路径）；**Fallback**：`onnx-graphsurgeon` 在 ONNX 图层做 surgery  |
| 理由           | 源码改造最干净（一次性，可 review）；graph surgery 易脆，跨 PyTorch/ONNX 版本可能失效                                                                  |
| 权衡           | 需维护 DINOv3 fork（含改造 patch）；但 patch 很小（< 20 行），可作为 PR 反馈到上游                                                                     |

### ADR-008 · TRT 版本锁定 *（V1.0.1 新增）*

| 项目           | 内容                                                                                                                                                                  |
| -------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| 决策点         | TRT 10.x 选哪个小版本                                                                                                                                                 |

### ADR-009 · 静态分辨率 + 动态 batch *（V1.0.1 新增，正式化 ADR-002 隐含决策）*

| 项目           | 内容                                                                                                         |
| -------------- | ------------------------------------------------------------------------------------------------------------ |
| 决策点         | 是否支持多分辨率引擎                                                                                         |
| **选择** | **静态分辨率（224×224）+ 动态 batch（B=1/4/8/16/32）单 profile**；多分辨率（336/518）作为独立引擎     |
| 理由           | 197 token 形状契约依赖固定分辨率；混合分辨率在单引擎中破坏 token 数恒等式；为 cudaGraph 优化（V1.1）铺平道路 |
| 权衡           | 多分辨率需多引擎，存储/管理成本上升                                                                          |
| 影响           | `engine_builder.py` profile 配置；`benchmark/` 矩阵按引擎档位拆分                                        |

## 6. 跨语言数值一致性策略

> 📍 **V1.0.1 修订**：MaxAbsErr 阈值改为分档；TF32 显式禁用。

| 措施                       | 实现                                                                                                                                                                                               |
| -------------------------- | -------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **共享预处理常量**   | `config/preprocess.yaml` 单一来源；Python/C++ 启动加载，禁止硬编码 mean/std                                                                                                                      |
| **位级一致性验证**   | 单测：同一 PNG 经两端预处理后 `MaxAbsErr < 1e-7`；CI 强制                                                                                                                                        |
| **统一 stream 语义** | 两端均用非默认 stream，调用结束 `cudaStreamSynchronize` 后再读出                                                                                                                                 |
| **确定性数值**       | TRT builder `kOBEY_PRECISION_CONSTRAINTS` + `kSTRICT_TYPES` + **`kDISABLE_TF32`（V1.0.1 新增）**；C++ 端 `CUBLAS_PEDANTIC_MATH`                                                      |
| **同一引擎二进制**   | Python 与 C++ 共享同一份 `.engine`，仅 runtime 不同；engine SHA256 写入 benchmark 报告                                                                                                           |
| **对齐验证流程**     | `tests/test_cross_lang_parity.py`：随机 10 张图 → Python 推理 → C++ 推理 → **逐输出**对比；FP32 阈值 `MaxAbsErr ≤ 1e-5`（TF32 disabled），FP16 `≤ 1e-3`，INT8 `cos_sim ≥ 0.99` |
| **种子与随机性**     | INT8 校准数据集顺序固定；预处理无随机 augmentation                                                                                                                                                 |

### 10.2 关键依赖链

- P2 依赖 P1：multi-layer hook + RoPE 改造决定 ONNX 导出图结构
- **M2.1（RoPE PoC）是 P1→P2→P3 全链路风险点**，必须前置验证
- P3 依赖 P2：4 输出 ONNX 是引擎构建输入
- P4 依赖 P3：FP16 通过后才做 INT8（共享构建脚手架与校准基础设施）
- P5 依赖 P3+P4：需所有精度引擎齐全
- P6 依赖 P5：选定最优引擎才封装 C++
- P7 横跨全程，但终稿依赖 P5/P6
- **关键路径**：P1（含 M2.1）→ P2 → P3 → **P4** → P5（INT8 + RoPE 是双瓶颈）
