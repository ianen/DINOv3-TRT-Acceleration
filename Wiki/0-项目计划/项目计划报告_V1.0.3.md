# DINOv3-TRT-Acceleration 项目计划报告 V1.0.3

> **状态**：**Proposed**（V1.0.3 规划文档，未实施）
> **触发**：V1.0.2 实测确认 single-inference latency 在 r518 b8 已 SM-saturated（3.40× cpp end-to-end 是 RTX 5080 物理上限 without QAT），5.0× 单 inference 旗舰目标 mathematically unreachable on TRT 10.13.2.6 + PTQ。用户明确指示 V1.0.3 切入"并行推理 + 充分利用全 GPU 性能"，要求先做工业界 reusable 方案调研再撰写计划。
> **范围**：Throughput-oriented serving（不再追 single-inference latency）；正交于 V1.0.2 已穷尽的 PTQ paradigm；不依赖 V1.3 QAT。
> **核心洞察**：V1.0.2 发现 r518 b8 单流已饱和 SM（multi-stream N=2 仅 1.023×）— V1.0.3 必须重新定义 metric：从"单 inference 多快"转为"GPU 单位时间能处理多少 inference"。

---

## 1. V1.0.2 留下的真实 gap

V1.0.2 实测结论（`TRT_acceleration_metrics_V1.0.2-delta.md`）：

| 指标 | r224 b1 (low load) | r336 b8 (medium) | r518 b8 (saturated) |
|---|---|---|---|
| Single-stream cpp latency speedup | 1.135× (Graphs) | — | **3.40× (V1.0.1 baseline)** |
| Multi-stream N=2 aggregate | **1.513×** ⭐ | 1.103× | **1.023×**（无效）|
| Multi-stream N=4 aggregate | 1.233× | 1.096× | 1.009× |

**Pattern**：V1.0.2 multi-stream 收益与 GPU utilization **单调反相关** — saturated regime 下完全无效。

**真实 gap**（V1.0.3 切入）：
1. ❌ 单流已饱和 → 单 inference 加速没空间
2. ✅ **聚合吞吐**未被严肃优化 — 现有 evaluate/benchmark 全是 sequential
3. ✅ **Dynamic batching** 未实施 — 业务侧多 request 没聚合
4. ✅ **Concurrent model instances** 未实施 — engine 只 load 一份
5. ✅ **Engine sharing** 未做 — V1.0.2 multi-stream 第一切是每 worker 独立 load engine（4× memory）

V1.0.3 = 把这 4 项已知 gap 用工业界 reusable 方案打通。

## 2. V1.0.3 SMART 目标（Throughput-oriented）

### 2.1 旗舰目标（headline）

**Aggregate throughput at sustained concurrent load 提升 ≥ 2.0× over V1.0.2 single-stream baseline**:

| Metric | V1.0.1/V1.0.2 baseline | **V1.0.3 target** | Stretch | Notes |
|---|---|---|---|---|
| **G1 r224 b1 aggregate qps** | 343.69 (V1.0.2 N=1) | **≥ 800 qps** (2.33×) | 1000 qps | dynamic batching + concurrent instances |
| **G2 r336 b8 aggregate qps** | 362.97 | ≥ 600 qps (1.65×) | 750 qps | medium load regime |
| **G3 r518 b8 aggregate qps**（旗舰点）| 156.46 | **≥ 200 qps** (1.28×) | 250 qps | saturation regime；有限增益 |
| **G4 dynamic batching r518 latency-throughput curve** | N/A | 完整 max_bs={1,4,8,16,32} 曲线 | — | paper §6 throughput limit |
| **G5 cos_min preservation** | 0.9977 | **≥ 0.997** | 0.999 | concurrent execution 必须 bit-exact |
| **G6 cross-tool comparison** | N/A | Triton vs custom pool ≤ 10% gap | 持平 | 验证 in-tree pool 实现质量 |
| **G7 GPU utilization 实测** | N/A | **SM ≥ 95% (saturation regime) / Tensor Core ≥ 70% (BF16 dense ceiling) / HBM ≥ 90% (b≥16 memory-bound regime) / power 280-340W (RTX 5080 TBP 360W, no throttle)** | TC > 75% | 数据驱动 V1.3 QAT 启动论证 |

> **G7 注脚**：BF16 dense path 的 Tensor Core 利用率结构性上限约 70%（INT8 / FP8 / 2:4 sparse Tensor Core 路径需 V1.3 QAT 才能解锁），G7 的 70% 即 V1.0.3 dense paradigm 内的合理 ceiling，超过 75% 视为 stretch。SM ≥ 95% 验证 r518 b8 已物理饱和（V1.0.2 ADR-015 N=2 1.023× 暗示 SM 已打满，G7 直接量化确认）；HBM ≥ 90% 验证 b≥16 memory-bound regime；power 280-340W sustained 是 RTX 5080 healthy load 区间（TBP default 360W / max 400W），power 不超过 360W 即无 power throttle，温度 < 83°C 即无 thermal throttle。
>
> **G7 saturation gate 已实证（2026-05-02 r518 b8 b1 single-stream）**：SM% mean **99.08%** / p50 99 / p95 100 / max 100 — saturation regime 物理上限达成；HBM% 20%（b=8 是 compute-bound，符合分类）；power avg 326.2W / peak 332.6W（headroom 28W to TBP）；temp peak 72°C（headroom 11°C to throttle）。详见 `Wiki/2-实验结果/V1.0.3-first-G7-datapoint_2026-05-02.md`。

## 3. 工业界可复用方案调研（reusability assessment）

5 个候选方案 × ViT-L/16 适用性：

| 方案 | 核心能力 | ViT-L 适用性 | V1.0.3 推荐度 | 资源 |
|---|---|---|---|---|
| **NVIDIA Triton Inference Server** | dynamic batching + concurrent model instances + multi-framework | ⭐⭐⭐⭐⭐ ViT 完全适用（任何 TRT engine 都能 host）| **A1 主推** | [docs](https://docs.nvidia.com/deeplearning/triton-inference-server/user-guide/docs/index.html) |
| **CUDA MPS** | Multi-process 共享 GPU context | ⭐⭐ small models 有效；ViT-L compute-bound 类似 7B+ regime（"sweet spots 外不显效"）| B1 评估 | [docs](https://docs.nvidia.com/deploy/mps/index.html) |
| **TensorRT C++ multi-context** | 1 shared engine + N IExecutionContext + N streams | ⭐⭐⭐⭐ 直接对应 ADR-015 §4.2 设计 | **A2 必做（继承 V1.0.2）**| [TRT C++ docs](https://docs.nvidia.com/deeplearning/tensorrt/latest/inference-library/c-api-docs.html) |
| **TensorRT-LLM** | in-flight batching + paged attention（KV cache 优化）| ⭐ ViT pure encoder 无 KV cache，paged attention 不适用 | C 不适用（文档化 negative）| [docs](https://nvidia.github.io/TensorRT-LLM/) |
| **vLLM** | continuous batching + paged attention（LLM 优化）| ⭐⭐ vLLM 0.5+ 加了 ViT CUDA graph 但其核心增益是 decode-phase；ViT 无 decode | C 不适用（文档化 negative）| [vLLM docs](https://docs.vllm.ai/) |

### 3.1 关键 finding

**Triton Inference Server** 是**ViT throughput-oriented serving 的 canonical reusable 方案**：
- "Running models concurrently and using dynamic batching of Triton led to almost a 70% increase in throughput" — NVIDIA 官方实测
- 直接接 TensorRT engine（项目已有 `dinov3_vitl16_4out.bf16.prefer.engine`）
- `instance_group` config 一行就启用 N concurrent instances
- `dynamic_batching` config 一段配置启用 request 聚合 + max_queue_delay_microseconds 调优
- 不重发明轮子

**CUDA MPS** 在 ViT-L compute-bound regime **预期收益小**：
- 工业实证："Outside [small model + short context] sweet spots... GPU-level benefits cannot be captured easily"
- ViT-L r518 b8 是 compute-bound（V1.0.2 ADR-015 实证 GPU 已饱和），属"sweet spots 外"
- 但 RTX 5080 Volta+ 架构原生支持 MPS hardware enhancement，可作 secondary 评估

**TensorRT-LLM / vLLM** 对 ViT 不适用 — 它们的核心增益来自 LLM decode 阶段的 KV cache 优化（paged attention / in-flight batching），而 ViT pure encoder 没有 generation loop。需文档化作 V1.0.3 ADR negative result。

### 4.1 ADR-019 — Triton Inference Server 集成（A1 主推）

**核心任务**：
1. Local 部署 Triton Inference Server（Docker 或 binary）
2. 把现有 `dinov3_vitl16_4out.bf16.prefer.engine` 装进 Triton model repository
3. 配置 `config.pbtxt`：
   - `instance_group { count: 2; kind: KIND_GPU }` — 2 concurrent instances
   - `dynamic_batching { preferred_batch_size: [4, 8]; max_queue_delay_microseconds: 1000 }` — 聚合 1ms 内的请求
4. 用 `perf_analyzer` 跑 throughput-latency 曲线（benchmark suite 现成）
5. 数据 push 到 `Code/Artifacts/reports/v103_triton_throughput_matrix.csv`

**验收**:
- G1 r224 b1 aggregate qps ≥ 800（vs V1.0.2 single-stream 343.69）
- G6 cos_min vs FP32 reference bit-exact（Triton 不改精度）
- `perf_analyzer` 报告作 paper §6 throughput section 主素材
- **G7 utilization 数据采集（必须）**：
  - `nvidia-smi --query-gpu=utilization.gpu,utilization.memory,memory.used,power.draw --format=csv -lms 100`
    跑 benchmark 全程，输出 `v103_triton_utilization_timeline.csv`
  - 单 inference 用 `ncu --metrics sm__throughput.avg.pct_of_peak_sustained_elapsed,tensor_active.avg.pct_of_peak_sustained_elapsed,dram__throughput.avg.pct_of_peak_sustained_elapsed` 采一次
  - benchmark CSV row 加 `sm_pct / tensor_core_pct / hbm_pct / avg_power_w` 4 列

**Out of scope**：分布式 serving / Kubernetes / 生产部署 — V1.0.3 仅 single-host 验证

### 4.2 ADR-020 — C++ Multi-Context Engine Pool（A2 自建）

**继承自 ADR-015 §4.2 未实施部分**。设计：

```
TRTInfererPool (新)
├─ 1 shared std::shared_ptr<ICudaEngine>
├─ std::vector<PerStreamState> N (复用 V1.0.2 PinnedBuffer + CudaGraphPool)
│   ├─ IExecutionContext (per stream)
│   ├─ cudaStream_t
│   ├─ PinnedBuffer (V1.0.2)
│   └─ CudaGraphPool entry (V1.0.2 LRU)
└─ submit(idx, input, outputs) → Status (round-robin / LRU dispatch)
```

**复用 V1.0.2 模块清单**：
- `Code/cpp/include/dinov3_trt/cuda_graph_pool.h` (194 行 LRU) — 100% 复用
- `Code/cpp/include/dinov3_trt/pinned_buffer.h` (47 行) — 100% 复用
- `Code/src/dinov3_trt/engine/trtexec.py` `additional_profiles` field — 100% 复用（multi-profile engine）
- `DINOV3_USE_CUDA_GRAPH` env var — 100% 复用

**新建文件**：
- `Code/cpp/include/dinov3_trt/trt_inferer_pool.h` (~100 行)
- `Code/cpp/src/trt_inferer_pool.cpp` (~300 行)
- `Code/cpp/tests/test_trt_inferer_pool.cpp` (~150 行 GoogleTest)
- `Code/cpp/bench/bench_stream_pool.cpp` (~180 行 benchmark binary)

**TRT 官方 best practices 编码进设计**（来自 web search）：
- Engine deserialize ONCE in main thread, share via `shared_ptr<ICudaEngine>` ✓
- 1 IExecutionContext per stream（contexts 不 thread-safe，TRT 文档明示）✓
- nonmodifying engine ops 是 thread-safe ✓
- CUDA context 在 process start 一次性创建，不 per-stream 创建 ✓

**验收**:
- G1 r224 b1 N=2 aggregate ≥ 700 qps（vs Python 520 qps，C++ 减少 GIL contention）
- G3 r518 b8 N=2 aggregate ≥ 200 qps（vs Python 160 qps）
- G6 vs Triton 差 ≤ 10%（验证 in-tree pool 工程质量）
- bit-exact 跨语言（C++ pool output vs Python TRT runtime）
- **G7 utilization 数据采集（必须）**：
  - C++ pool benchmark 全程跑 `nvidia-smi --query-gpu=utilization.gpu,utilization.memory,memory.used,power.draw --format=csv -lms 100`，输出 `v103_cpp_pool_utilization_timeline.csv`
  - 跑 N=1/2/4 三种 pool size 各采一次 `ncu --metrics sm__throughput..., tensor_active..., dram__throughput...`，验证多 stream 实际是否提升 SM 占用率
  - **关键验证**：V1.0.2 ADR-015 实证 r518 b8 N=2 仅 1.023× — G7 数据应直接确认 N=1 时 SM_pct 已 ≥ 95%（saturation 已达，多 stream 无空间），同时验证 r224 b1 low-load regime 多 stream 真填了 idle SM
  - benchmark CSV row 加 `sm_pct / tensor_core_pct / hbm_pct / avg_power_w` 4 列

### 4.3 ADR-021 — Dynamic Batching Tuning Matrix

**前置依赖**：ADR-019 Triton 部署完成。

**任务**：用 `perf_analyzer` 跑 grid search:
- max_batch_size ∈ {1, 4, 8, 16, 32}
- preferred_batch_size ∈ subset of [1..max_batch_size]
- max_queue_delay_microseconds ∈ {0, 100, 500, 1000, 5000}
- concurrent_clients ∈ {1, 2, 4, 8, 16}

**输出**：完整 throughput-vs-latency Pareto frontier（paper §6 thesis statement）。

**关键 finding 期望**：dynamic batching 在 r518 b16 解锁 V1.0.1 memory-bound 边界（V1.0.1 b≥16 only 1.0× speedup，dynamic batching 通过 throughput 视角避开此 wall）。

### 4.4 ADR-022 — CUDA MPS 评估（P2 secondary）

**任务**：测 MPS 启用 vs disabled 下，多 client 并发推理的 aggregate qps。

**预测结果**：ViT-L compute-bound regime 收益 < 5%（基于 industry 实证 "outside sweet spots cannot capture easily"）。如果实证为 negative，文档化作 paper §6 limitations 素材；如 ≥ 10% 增益则进 ship 路径。

### 4.5 ADR-023 — TensorRT-LLM / vLLM Negative Documentation

**任务**：明确文档化为何这两个方案 **不适用** ViT pure encoder：
- TensorRT-LLM: in-flight batching 优化的是 KV cache 跨 token；ViT 单次 forward 无 cache
- vLLM: paged attention 优化 decode 内存；ViT 无 generation loop
- 但 vLLM 0.5+ 加了 ViT CUDA graph capture（已被 V1.0.2 ADR-012 cover）

**输出**：1 page ADR 文档作 paper §7 future work 中"为何不走 LLM 路线"的 reference。

### 6.1 GPU utilization 监控工具栈（实施协议）

ADR-019（Triton path）与 ADR-020（C++ pool path）必须遵循同一采集协议。集中文档于此避免散落。

**工具栈**：
| 工具 | 粒度 | 用途 |
|---|---|---|
| `nvidia-smi --query-gpu=...` | 100ms cadence | benchmark 全程时序，主指标采集 |
| `ncu` (Nsight Compute) | per-kernel | 单点采样 SM / Tensor Core / HBM peak utilization |
| `trtexec --profilingVerbosity=detailed --exportProfile` | per-layer | kernel time breakdown，定位瓶颈层 |

**采集 cadence**：
- benchmark 全程 nvidia-smi 时序（warm-up 前 5 秒数据丢弃）
- 每个 (resolution, batch, n_streams) 组合做一次 ncu 5-second 取样
- 单 inference baseline 跑 1 次 trtexec --exportProfile（V1.0.2 ADR-018 已有 attention 26.4% 数据）

**聚合规则**：
- nvidia-smi raw timeline → mean / p50 / p95 / max per (resolution, batch, n_streams) → 写入主 benchmark CSV 的 4 个新列
- ncu 单点数据归档 `Code/Artifacts/reports/utilization/<run_id>/ncu_metrics.txt`

**数据归档结构**：
```
Code/Artifacts/reports/utilization/
├── <run_id>/
│   ├── timeline.csv              (nvidia-smi raw 100ms cadence)
```

## 13. 与 V1.3 QAT 的关系

V1.0.3 与 V1.3 QAT **正交**：
- V1.0.3 改 serving runtime（单 inference 时延不变；改 throughput）
- V1.3 改 model weights（重训降 INT8 误差）
- 联合后期望：V1.3 INT8 cos_min 0.99+ R1 PASS（突破 V1.0.2 PTQ wall）+ V1.0.3 Triton dynamic batching → INT8 模型在 throughput 维度再叠加增益
- 互不阻挡，可并行启动

## 14. 一句话总结

**V1.0.3 旗舰目标 = aggregate throughput ≥ 800 qps @ r224 b1 / ≥ 200 qps @ r518 b8（保 cos_min ≥ 0.997）**。从 V1.0.2 实证的 GPU saturation 边界出发，**重新定义 metric 从 single-inference latency 转为 concurrent serving throughput**。主路径 = 集成 NVIDIA Triton Inference Server（ADR-019）+ 自建 C++ 多 context pool（ADR-020 继承 ADR-015 §4.2）；6 周 wallclock，复用 V1.0.2 全部基础设施（CudaGraphPool / PinnedBuffer / additional_profiles），不依赖 V1.3 QAT。
