# M1 正式结果摘要（2026-04-30）

## 结论

当前有效候选是 **BF16 prefer TensorRT engine**：

- 正式 FP16 engine 虽然速度快，但在正式 HF 权重上输出全 NaN，不能作为有效候选。
- BF16 prefer engine 在真实图片 eval、Python TensorRT runtime、C++ TensorRT runtime 中均输出有限值，并相对 FP32 有稳定加速。
- 默认 ModelOpt INT8 Q/DQ 即使用 500 张真实图片校准也仍然塌缩，不能构建为候选结果。
- 节点级 partial INT8 能避免塌缩，但速度慢于 BF16 prefer，只保留为敏感性分析证据。

## 性能结果

### trtexec GPU Compute

该组数据在 RTX 5080 上使用 `nvidia-smi -lgc 2752,2752` 锁定 graphics clock，并启用 `trtexec --useSpinWait`，更适合作为正式报告中的可复现性能口径。

已新增 `formal_benchmark_matrix.csv/.json/.md`，作为 P5 benchmark 表的当前机器可读入口。当前 matrix 共 38 行，覆盖 224 分辨率下 BF16 prefer、partial INT8、`layer19`、`layer19_attention` 在 locked `trtexec` 与 C++ runtime 口径中的关键对比；完整 336/518 分辨率仍待补测。

BF16 prefer 相对 FP32：

| batch | FP32 median ms | BF16 median ms | latency speedup | throughput speedup |
|---:|---:|---:|---:|---:|
| 1 | 7.0352 | 2.8672 | 2.45x | 2.37x |
| 4 | 15.860 | 6.218 | 2.55x | 2.44x |
| 8 | 28.322 | 10.075 | 2.81x | 2.65x |
| 16 | 57.696 | 18.757 | 3.08x | 2.90x |
| 32 | 120.000 | 36.968 | 3.25x | 3.07x |

### C++ Runtime End-to-End

该路径包含 H2D、enqueue、D2H、stream sync，更接近生产封装延迟。

| batch | FP32 median ms | BF16 median ms | latency speedup | throughput speedup |
|---:|---:|---:|---:|---:|
| 1 | 7.46110 | 3.28485 | 2.27x | 2.27x |
| 8 | 30.0546 | 12.1829 | 2.47x | 2.46x |
| 32 | 126.804 | 44.7938 | 2.83x | 2.83x |

结论：BF16 prefer 在 trtexec 与 C++ runtime 中都稳定快于 FP32，C++ runtime 侧 batch 越大加速越明显。

## Python/C++ 跨语言一致性

新增 `dinov3_trt_dump_outputs.exe` 将 C++ runtime 的四个输出 tensor 写成 float32 binary，再由 `scripts/compare_cpp_python_parity.py` 使用同一 deterministic sine input 跑 Python TensorRT runtime，并逐输出比较全量 tensor。

batch 1 结果：

| engine | output | max abs error | RMSE | cosine |
|---|---|---:|---:|---:|
| FP32 | `feat_layer_4/12/16/20` | 0 | 0 | 1 |
| BF16 prefer | `feat_layer_4/12/16/20` | 0 | 0 | 1 |
| partial INT8 | `feat_layer_4/12/16/20` | 0 | 0 | 1 |

结论：当前 Python 与 C++ runtime 在同一 TensorRT engine、同一输入、同一输出 buffer 规约下为 bit-identical，满足 V1.0.1 的 G3 跨语言一致性要求。

## FP16 负例

正式 FP16 engine 不能作为结果报告中的有效候选：

- 默认 FP16 在正式权重下可 build、可 benchmark，但 Python TensorRT runtime 输出全 NaN。
- 分 block FP32 诊断显示，需要大范围 FP32 回退才能避免 NaN，性能随之接近 FP32。
- 因此 FP16 只能作为“速度但不正确”的负例列入报告，不能和 BF16/FP32 做同级有效候选比较。

## INT8 结论

默认 ModelOpt INT8 Q/DQ 路线当前不可用：

- 2 张 smoke calibration 会塌缩。
- 500 张真实 calibration 仍塌缩。
- 失败已在 ONNX Runtime 层复现，不是 TensorRT engine build 独有问题。

定位结果：

- `MatMul-only` 不零范数，但整体 cosine 不达候选门限。
- `MatMul+Add` 直接复现前三层零范数。
- `MatMul+LayerNormalization` 直接复现前三层零范数。
- `MatMul+Mul` 不零范数，但 cosine 大幅劣化。
- 只量化 `/model/layer.16` 到 `/model/layer.19` 的 MatMul 可控，但只影响 `feat_layer_20`，属于 partial INT8。

partial INT8 在 1000 张真实图片上的结果：

| output | cosine mean | cosine min | max abs error | RMSE |
|---|---:|---:|---:|---:|
| `feat_layer_4` | 1.000000 | 1.000000 | 0.000148 | 0.000000 |
| `feat_layer_12` | 1.000000 | 1.000000 | 0.000421 | 0.000002 |
| `feat_layer_16` | 1.000000 | 1.000000 | 0.001883 | 0.000008 |
| `feat_layer_20` | 0.989177 | 0.988792 | 36.703 | 1.49067 |

locked trtexec 性能结论：

- trtexec 相对 FP32 仅 `1.18x / 1.22x / 1.22x`。
- trtexec 相对 BF16 prefer 仅 `0.48x / 0.43x / 0.38x`。
- C++ runtime 相对 FP32 仅 `1.17x / 1.23x / 1.22x`。
- 相对 BF16 prefer 明显更慢。

后续 MatMul block sweep 进一步缩小量化范围：

| variant | ONNX 32-image `feat_layer_20` cosine mean | TRT 1000-image `feat_layer_20` cosine mean | locked trtexec speedup vs FP32 | speed vs BF16 prefer |
|---|---:|---:|---:|---:|
| layers16-19 | 0.988997 | 0.989177 | 1.18x / 1.22x / 1.22x | 0.48x / 0.43x / 0.38x |
| layers17-19 | 0.990877 | 未构建 TRT | 未测 | 未测 |
| layers18-19 | 0.993062 | 未构建 TRT | 未测 | 未测 |
| layer19 | 0.995601 | 0.995659 | 1.05x / 1.07x / 1.06x | 0.43x / 0.38x / 0.33x |
| layer19_attention | 0.998970 | 0.998994 | 1.04x / 1.05x / 1.04x | 0.42x / 0.37x / 0.32x |
| layer19_mlp | 0.996596 | 未构建 TRT | 未测 | 未测 |

C++ runtime follow-up 显示，`layer19` 单层 INT8 的 Python/C++ 输出仍为 bit-identical，但端到端速度相对 FP32 只有 `1.07x / 1.08x / 1.06x`，相对 BF16 prefer 只有 `0.47x / 0.43x / 0.37x`。

结论：缩小到 `layer19` 或 `layer19_attention` 后，INT8 正确性可以回到 0.99 以上，但加速收益几乎消失，且明显慢于 BF16 prefer。因此 partial INT8 只作为敏感性分析和后续节点级量化路线证据，不作为当前部署候选。

## 报告使用建议

正式报告中建议采用以下叙述：

1. FP32 是正确性与性能基线。
2. FP16 是 Blackwell + TRT 10.13 下的失败负例，展示 NaN 风险。
3. BF16 prefer 是当前有效候选，兼具真实图片稳定性、Python/C++ 可执行性和明确加速。
4. INT8 默认 Q/DQ 仍不可用，问题来自量化图/校准配置本身；节点级 partial INT8 证明可控，但 layer19 单层结果显示正确性和速度存在强 tradeoff。
5. 完整 ImageNet val 到位后，替换 Imagenette manifest 并重跑 `formal_summary`，不改变评估脚本和报告结构。
