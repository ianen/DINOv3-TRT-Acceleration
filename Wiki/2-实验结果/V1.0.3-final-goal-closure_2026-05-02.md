## Key Architectural Findings (paper-ready material)

### 1. Phase 2 shared-engine multi-context architecture works

Originally V1.0.3 ADR-020 Phase 1 (per-slot independent engine deserialization) crashed at concurrent N=2 with TRT 10.13 Myelin runner.cpp:778 errors. The first instinct was "TRT 10.13 is not thread-safe."

Empirical investigation contradicted this:
- **Multi-process** (2 separate OS processes, each with own engine) — works (1.087× speedup, low due to inter-process CUDA context contention)
- **Phase 2 prototype** (`test_concurrent_contexts.cpp` — single shared engine, 2 independent IExecutionContext + cudaStream + concurrent enqueueV3) — **works perfectly**, 1.515× at r224 b1 N=2, 2.683× at r224 b8 N=2 (922 qps pure compute)

So the Myelin error was caused by **duplicate engine deserialization in the same process**, not fundamental TRT thread-safety. NVIDIA's documentation that "single engine, multiple IExecutionContexts on different streams = thread-safe" is correct. ADR-020 Phase 2 (production rewrite) implements this and ships successfully.

### 2. BF16 dense path is empirically saturated across all tested regimes

V1.0.3 G7 utilization measurement provides the first comprehensive empirical evidence that the BF16 dense compute path is at the GPU's physical ceiling for our model:

| Regime | SM% mean | SM% p95 | Headroom |
|---|---|---|---|
| r518 b8 N=1 | **99.08%** | 100% | none — physical ceiling |
| r336 b8 N=1 | **96.39%** | 99% | ~3% (negligible) |
| r224 b1 N=1 | 88.24% | 92% | ~12% (only regime where multi-context helps) |
| r224 b1 N=2 | **95.77%** | 99% | filled by multi-stream (Δ +7.5 pp) |

This is the empirical evidence that V1.3 QAT (the only quantization path that preserves cos_min ≥ 0.99 — V1.0.2 PTQ frozen-negative across 5 vectors) is the **sole remaining acceleration avenue**. Dense path is out of headroom on this hardware/model.

### 3. G2 / G3 plan thresholds were over-set

The V1.0.3 plan §2.1 set:
- G2 r336 b8 ≥ 600 (1.65× over baseline 363) — required ≥ 96% SM lift, but N=1 already at 96.39%
- G3 r518 b8 ≥ 200 (1.28× over baseline 156) — required ≥ 28% lift, but N=1 already at 99.08% SM

Both thresholds assumed multi-context could provide their needed speedups. The G7 saturation data, combined with V1.0.2 ADR-015 r518 b8 N=2 = 1.023× empirical evidence, shows these were **physically unreachable on this hardware/engine via in-process multi-context alone**. Achieving them would require:

- A **smaller / quantized model** (V1.3 QAT) — reduces per-inference compute, restoring headroom for multi-context to fill
- **Inter-request batching** (Triton dynamic_batching) — aggregates multiple single-image requests into one larger batch, amortizing fixed overheads
- **Multi-GPU** (out of scope; RTX 5080 single-card)

The plan analysis was sound but the SMART thresholds did not account for the SM saturation regime that V1.0.2 ADR-015 had already partially documented (1.023× at r518 b8 was a clear signal). V1.0.3 G7 has now made this regime fully quantitative.

### 4. G1 81% achieved end-to-end; 100% prototype upper bound

The Phase 2 prototype `test_concurrent_contexts.cpp` reached 922 qps at r224 b8 N=2 (2.68×) — **above the 800 plan target**. This is "pure compute" throughput (no H2D / D2H / cudaStreamSynchronize wallclock).

The production pool with full host-device transfer overhead reaches **720 qps (2.094×)**. The 26% gap between prototype and production = per-call host transfer + sync overhead, which:
- **Pinned worker buffers** (`--pinned` flag): only +1.4% gain at this workload size; the cudaMallocHost overhead nearly cancels the DMA savings
- **Caller pre-pinning the API surface** (Phase 3 API change): not implemented; estimated to recover most of the 26% but at API redesign cost
- **Triton IPC shared-memory**: completely sidesteps host transfer; this is the path the V1.0.3 plan assumed when setting G1 = 800

Net: **V1.0.3 in-process pool ceiling = ~700 qps end-to-end**. Hitting the plan 800 qps requires either a Triton install (Docker user-blocker) or a Phase 3 pinned-buffer API redesign.
