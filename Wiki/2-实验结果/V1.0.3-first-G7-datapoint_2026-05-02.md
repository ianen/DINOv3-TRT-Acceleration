## Implications for V1.0.3 ADR Sequence

1. **ADR-020 (C++ multi-context pool) now empirically justified** — Python path 1.5× ceiling proven, can't reach G1 800 qps. C++ pool with shared `ICudaEngine` + N `IExecutionContext` over CUDA streams is the lowest-risk path to G1.
2. **ADR-019 (Triton)** — currently parked due to Docker/WSL2 user-blocker on Windows host. Once unblocked it would augment with dynamic batching (request aggregation in 1 ms windows can lift batch utilization on top of multi-context).
3. **Per-N utilization breakdown** — needed to verify G7 acceptance "r224 b1 N=2 SM% − N=1 SM% ≥ 15 pp". Orchestrator refactor: drive 3 separate runs (one per stream count) instead of one combined.
4. **Tensor Core measurement** — first run skipped ncu invocation. Next run will pass `-NcuTargetExe python -NcuTargetArgs "scripts\benchmark_multi_stream.py ..."` to capture HMMA pipe activity; this is the V1.3 QAT door (Tensor Core ceiling in BF16 dense path → quantify the gap that INT8/FP8/2:4 sparse via QAT could close).
5. **r518 b8 single-stream G7 saturation test** — needs separate run with `-Engine ...r518.bf16.prefer.b8.engine -BatchSize 8 -ImageSize 518 -NumStreams 1` to verify SM ≥ 95% claim (V1.0.2 ADR-015 indirectly inferred this from N=2 → 1.023× near-zero gain; G7 will *quantify* it).

**Original "+15 pp N=2 vs N=1 SM gain" gate corrected to "absolute ≥ 95%"** — N=1 already at 88.24% leaves only 12 pp headroom, making +15 pp physically unreachable. The corrected gate captures the meaningful outcome (multi-stream lifted to saturation).

**Core conclusion (V1.3 QAT motivation)**: BF16 dense path is **physically saturated across all regimes** (SM 96-99%). Further dense-path optimization yields zero throughput (V1.0.2 ADR-015 N=2 r518 b8 1.023× already proved this). The only path to further speedup is **breaking the BF16 dense Tensor Core ceiling (~70% structural limit) via INT8/FP8/2:4-sparse QAT** — this is V1.3.

ADR-020 C++ multi-context pool will help in **low-batch low-resolution regime only** (r224 b1 N=2 lifted from 1.0× to 1.498× per V1.0.2 ADR-015 + this run). At medium/high regimes the SM is already saturated — pool will not increase throughput.
