## Abstract（IMRaD-form, ~250 words）

**Background**: Modern vision foundation models such as DINOv3 ViT-L/16 LVD-1689M expose multi-scale intermediate features for dense prediction downstream tasks (depth, segmentation), but FP32 inference cost prohibits deployment on consumer-grade hardware.

**Purpose**: We empirically map the post-training quantization (PTQ) boundaries of ViT-L/16 on NVIDIA Blackwell sm_120 with TensorRT 10.13, and establish whether mixed-precision strategies can break the upstream quantization noise barrier.

**Method**: We benchmark 12 inference precision candidates (FP32 / BF16 / FP16 / FP8 / INT8 PTQ + node-level partial / SmoothQuant α-sweep / 3 mixed-precision strategies) across resolutions {224, 336, 518} on RTX 5080. Cosine similarity against FP32 baseline is measured on 1000 real images per (resolution, candidate) pair. Cross-language Python ↔ C++ runtime parity is verified on bit-identical engine binaries.

**Findings**: BF16 prefer is the only candidate inside the (cos ≥ 0.99 ∧ speedup ≥ 2.2×) target region, achieving 3.86× peak speedup at r518 batch 8. Three independent mixed-precision tool-chains (PyTorch ModelOpt `disable_quantizer`, TensorRT `--layerPrecisions`, ONNX-level Q/DQ stripping) produce numerically equivalent results (cos_min within 0.0005 across paths), proving that the precision bottleneck is not in the chosen tool but in upstream cumulative INT8 quantization noise from blocks 0-15.

**Implications**: Mixed-precision PTQ is fundamentally insufficient for tight cosine constraints on this model + hardware combination. Quantization-aware fine-tuning (QAT) is the only remaining path; we publish a complete design ADR for V1.3 implementation.

**Keywords**: TensorRT, ViT, post-training quantization, mixed-precision, DINOv3, Blackwell

## 1. Introduction

### 1.1 Context and Background

- **2-3 paragraphs**：DINOv3 (Meta, 2024) as visual self-supervised foundation model on LVD-1689M; multi-scale intermediate features power dense prediction via DPT-style fusion; deployment cost on consumer GPUs.
- Citations: DINOv3 paper, DPT paper, TensorRT documentation, NVIDIA Blackwell architecture white paper.

### 1.2 Problem Statement

- ViT-L/16 FP32 inference cost on RTX 5080 (Blackwell sm_120, 16 GB VRAM): **batch 8 r224 = 28 ms**.
- Project goal: reduce inference cost by ≥ 2× while preserving feat_layer_20 cosine ≥ 0.99 (a stringent precision requirement set by the deployment team to ensure downstream task fidelity).

### 1.3 Research Gap

- **Public TensorRT benchmark literature** for ViT-L on Blackwell sm_120 + TRT 10.13 is sparse (the architecture is new in 2026).
- **INT8 PTQ studies on ViT-L** historically focus on cos ≥ 0.95 thresholds; cos ≥ 0.99 with ≥ 2× speedup constraint creates a narrow operating region most studies don't probe.
- **Mixed-precision PTQ literature** typically tests one tool chain; cross-tool-chain equivalence has not been established empirically.

### 1.4 Purpose and Research Questions

The purpose of this study is to **empirically map the (precision, speedup) operating region** of TensorRT 10.13 inference for DINOv3 ViT-L/16 on Blackwell sm_120, and **identify the binding constraint** that prevents any INT8 PTQ candidate from entering the target region.

The following research questions guide this study:

1. **RQ1**: Among standard precision candidates (FP32, BF16, FP16, FP8, INT8 default + partial + SmoothQuant), which fall inside the target region (cos ≥ 0.99 ∧ speedup ≥ 2.2×) on this model + hardware combination?
2. **RQ2**: Does the choice of tool chain (PyTorch ModelOpt vs TensorRT command-line vs ONNX library) materially affect the precision-speedup trade-off of mixed-precision strategies that force a subset of transformer blocks back to higher precision?
3. **RQ3**: What empirical evidence determines the binding constraint for failed candidates, and what implementation path could lift it?

### 1.5 Significance of the Study

- **Empirical map** of new hardware-software intersection (Blackwell + TRT 10.13 + ViT-L) for community reference.
- **Cross-tool-chain equivalence proof** eliminates "wrong tool" hypothesis space, focusing future research on QAT.
- **Open reproducibility infrastructure**: 56-row machine-readable benchmark matrix + 8 figures + 271 unit tests + atomic SHA256 manifest.

## 3. Methodology

### 3.1 Hardware Setup

- RTX 5080, Blackwell sm_120, 16 GB VRAM, locked at 2752 MHz.
- TensorRT 10.13.2.6, CUDA 12.8, Windows 10.
- Locked-clock + `trtexec --useSpinWait` to suppress Windows WDDM jitter.

### 3.2 Model and Contract

- DINOv3 ViT-L/16 LVD-1689M (`facebook/dinov3-vitl16-pretrain-lvd1689m`); 24 transformer blocks; 4 register tokens (default API drops them, project main path uses 197-token contract `[B, 197, 1024]`).
- Output binding: 4 intermediate features `feat_layer_{4, 12, 16, 20}` (1-based; 0-based indices `(3, 11, 15, 19)`).

### 3.3 ONNX Export and RoPE Source-Patch

- HuggingFace Transformers `AutoModel.from_pretrained()` → `torch.onnx.export(opset=19)` with RoPE source-level patch eliminating ONNX `If` nodes (replacing `angles.tile(2)` with `torch.cat`) — ADR-007 documents this in detail.

### 3.4 Multi-Resolution Engine Strategy

- Per-resolution static spatial + dynamic batch profile (ADR-002, ADR-009).
- r518 batch 8: dual profile strategy (`min=1, opt=4, max=8` + separate `max=8` engine).
- Independent timing cache per (resolution, batch) pair.

### 3.5 Precision Candidates Evaluated

| Candidate | Tool chain | Description |
|---|---|---|
| FP32 | TRT default | Baseline |
| BF16 prefer | `--bf16` | Primary low-precision candidate |
| FP16 | `--fp16` | Tested as negative (NaN under formal weights) |
| FP8 PTQ (default) | ModelOpt FP8 | Aggressive |
| FP8 partial layer19 | ModelOpt + node allow-list | Sensitivity |
| INT8 default | ModelOpt INT8 explicit Q/DQ | Standard PTQ |
| INT8 partial layers16-19 / layer19 / layer19_attention | ModelOpt + allow-list | Sensitivity |
| INT8 SmoothQuant α=0.5/0.7/0.8 | ModelOpt SmoothQuant | Activation smoothing |
| INT8 SmoothQuant + skip 16-19 | ModelOpt `disable_quantizer` | Mixed-precision via PyTorch |
| INT8 SmoothQuant α=0.8 + trtexec `--layerPrecisions=l16-19:fp32` | TRT command-line | Mixed-precision via TensorRT |
| INT8 SmoothQuant α=0.8 + ONNX-level Q/DQ stripping (V1.2) | onnx-graphsurgeon-equivalent | Mixed-precision via ONNX |

### 3.6 Cosine Evaluation Protocol

- Imagenette2-320 val (1000 images, mutually exclusive with 500-image SmoothQuant calibration).
- Per-(image, output) cosine similarity vs FP32 baseline.
- Aggregation: cos_mean and cos_min per output across the 1000 images.
- Note: ImageNet val 50K is gated (HF 403); Imagenette serves as proxy distribution (full discussion in Limitations).

### 3.7 Cross-Language Parity Methodology

- Deterministic sine input (reproducible bit-identical input).
- Same engine binary loaded by Python TensorRT runtime and C++ runtime (MSVC + CUDA + RAII engine wrapper).
- Per-output max_abs_error + RMSE + cosine reported.

### 3.8 Pure-Python Testing Pattern (Methodological Innovation)

- ONNX graph manipulations (Q/DQ identification, classification, strip planning) split into:
  1. Pure-Python data-projection helpers (no `onnx` dependency) for identification + planning.
  2. Thin remote-only driver scripts for actual graph mutation.
- Coverage: 271 pytest cases, 111 source files, all unit-testable on developer macOS without GPU or native ONNX.

## 4. Results

### 4.1 BF16 Prefer Acceleration Results (RQ1)

**Figure 1**: `figures/benchmark_trtexec_bf16_speedup.svg` (multi-resolution speedup bar chart).

**Table 1**: BF16 prefer trtexec GPU-compute median speedup vs FP32:

| Resolution | b1 | b8 | b32 |
|---|---:|---:|---:|
| 224 | 2.45× | 2.81× | 3.25× |
| 336 | 2.80× | 2.96× | 3.25× |
| 518 | 3.12× | **3.86×** | — (16 GB cap) |

**Table 2**: C++ end-to-end speedup (includes H2D + enqueue + D2H + stream sync):

| Resolution | b1 | b8 |
|---|---:|---:|
| 224 | 2.27× | 2.50× |
| 336 | 2.60× | 2.85× |
| 518 | 2.83× | **3.40×** |

### 4.2 BF16 Prefer Cosine Results (RQ1)

**Figure 2**: `figures/benchmark_bf16_cosine_min.svg` + `figures/benchmark_bf16_cosine_mean.svg`.

**Table 3**: feat_layer_20 cos_min on 1000 Imagenette images:

| Resolution | cos_min | cos_mean |
|---|---:|---:|
| 224 | 0.998749 | 0.999127 |
| 336 | 0.998493 | 0.99936 |
| 518 | **0.999171** | 0.999721 |

Counter-intuitive finding: r518 (deepest hook on largest input) has the highest cos_min, attributable to BF16 quantization noise being diluted across more patch tokens (1025 vs r224's 197).

### 4.3 INT8 Sensitivity Trade-off Map (RQ1)

**Figure 3 (showcase)**: `figures/benchmark_bf16_vs_int8_tradeoff.svg` — 12-point scatter, X = cosine_mean, Y = b8 latency speedup vs FP32, G2 ideal region (cos ≥ 0.99 ∧ speedup ≥ 2.2×) shaded green.

**Key observation**: BF16 prefer is the only point inside G2 ideal region. SmoothQuant α=0.8 best INT8 (cos 0.982 / speedup 3.48×) is closest but cos_min 0.968 falls 0.022 short of threshold.

### 4.4 Three-Tool-Chain Mixed-Precision Convergence (RQ2)

**Table 4**: Mixed-precision tool-chain comparison (forces blocks 16-19 to non-INT8):

| Tool chain | feat_layer_20 cos_min | b8 speedup vs FP32 |
|---|---:|---:|
| Full SmoothQuant α=0.8 (baseline) | 0.968 | 3.48× |
| ModelOpt `disable_quantizer` skip 16-19 | 0.971 (+0.003) | 2.41× (-30%) |
| trtexec `--layerPrecisions=l16-19:fp32` | 0.9683 (≈) | 3.43× (≈) |
| **ONNX-level Q/DQ stripping (V1.2)** | **0.9705** | **2.39×** |

**Convergence proof**: cos_min within 0.0005 across all three independent paths; speedup within 0.02× when both paths force fp32 fallback. Tool chain choice is empirically demonstrated to be irrelevant.

### 4.5 4-Layer Hook Selection Ablation

**Figure 4**: `figures/layer_ablation_diversity_vs_balance.svg` (X = mean inter-output cosine, Y = log10 magnitude max/min ratio).

**Table 5**: DPT-style 4-layer hook ablation (1000 images, PyTorch hooks):

| Candidate | layers (1-based) | mean inter-output cos | max/min magnitude ratio |
|---|---|---:|---:|
| project (selected) | 4/12/16/20 | 0.383 | **12.6×** |
| dpt (paper recommended) | 5/11/17/23 | **0.299** | 31.9× |
| late-heavy | 6/12/18/24 | 0.339 | 84× |

The project's choice provides tightest magnitude balance at the cost of slightly higher inter-output cosine, avoiding the late-heavy variant's 84× imbalance where the deepest hook would dominate fusion.

### 4.6 Cross-Language Parity (Sanity Check)

All 12 cross-language parity reports (224/336/518 × FP32/BF16-prefer + r224 partial-INT8 + layer19 INT8) show 4-output `max_abs_error = 0`, RMSE = 0, cosine = 1.0 against deterministic sine input.

## 5. Discussion

### 5.1 Root Cause Analysis (RQ3)

**Empirical observation**: V1.0+V1.1+V1.2 across 3 tool chains all produce equivalent mixed-precision results, all failing G2 cos_min by ~0.022.

**Mechanism**: blocks 0-15 INT8 PTQ accumulates ~10⁻² quantization error in feat_layer_15 output; this contaminated input propagates through blocks 16-19 unrecoverably even with fp32 internal computation. Mixed-precision in the **last 4 blocks** addresses the wrong locus — the precision bottleneck is **upstream cumulative noise**, not the last-stage computation.

### 5.2 Why Three Tool Chains Converge

All three implementations end up triggering the same TensorRT FP32 fallback for the un-quantized blocks. The PyTorch / TRT / ONNX layer doesn't matter because the underlying TRT runtime has identical compiled kernels regardless of how the un-quantized region was specified.

### 5.3 Implications for QAT (V1.3 Future Work)

- Quantization-aware fine-tuning is the only path to reduce upstream noise at the **weight level** (not just the activation calibration level).
- ADR-011 specifies a complete V1.3 design with 4-condition launch threshold (data, hardware, time, downstream baseline).
- Expected outcome: cos_min ≥ 0.99 + speedup ≥ 3.0× → first INT8 candidate inside G2 ideal region.

### 5.4 Methodological Innovations

- **Pure-Python testing pattern** for ONNX graph manipulation enables GPU-free dev workflow (271 tests on macOS).
- **Bidirectional remote-sync** with `--pull-reports` separates large binary artifacts from text-only reproducible artifacts.
- **Atomic SHA256 manifest with self-exclusion** resolves the "manifest-generates-itself" 0-byte file pre-creation bug.
- **Three-tool-chain convergence proof** as a methodological pattern for negative-result validation in inference systems research.

### 5.5 Comparison to Related Work

- vs. Standard ViT INT8 PTQ literature: most studies report cos ≥ 0.95 thresholds; our cos ≥ 0.99 stringent constraint exposes a regime where standard PTQ is insufficient.
- vs. SmoothQuant paper: validates SmoothQuant's activation-only smoothing improves cos by ~0.014 over default PTQ but does not cross 0.99 on ViT-L/16.
- vs. DPT paper: empirically validates and slightly modifies DPT's recommended layer choice (`[5,11,17,23]` → `[4,12,16,20]`) on ViT-L/16 for magnitude balance.

## 6. Limitations

### 6.1 Dataset Proxy

- ImageNet val 50K is gated on HuggingFace; Imagenette2-320 (10 classes, 13K images) used as proxy. While Imagenette is a strict ImageNet subset, smaller class diversity may underestimate worst-case PTQ error.

### 6.2 Single Hardware

- All measurements on RTX 5080 (Blackwell sm_120). Ada Lovelace (sm_89, no FP8 hardware) and Hopper (sm_90) may invert FP8 vs INT8 trade-offs.

### 6.3 TensorRT Version

- BF16 + Q/DQ Myelin Fill incompatibility and `--layerPrecisions` no-op behavior under explicit Q/DQ are specific to TRT 10.13.2.6; future versions may close these gaps.

### 6.4 QAT Not Implemented

- Four launch conditions (dataset unblock, training resources, time budget, downstream task baseline) are not simultaneously met; we provide V1.3 design ADR but defer implementation.

## 7. Conclusion

We empirically map the (precision, speedup) operating region of TensorRT 10.13 inference for DINOv3 ViT-L/16 on RTX 5080 (Blackwell sm_120) and demonstrate that **BF16 prefer is the only candidate in the (cos ≥ 0.99 ∧ speedup ≥ 2.2×) region**. Three independent mixed-precision tool chains (PyTorch ModelOpt, TensorRT command-line, ONNX library) converge to numerically equivalent results, proving that the precision bottleneck is upstream cumulative INT8 quantization noise rather than tool-chain choice. Quantization-aware fine-tuning (V1.3 ADR-011) is identified as the remaining path; we publish a complete reproducibility kit comprising 56-row benchmark matrix, 8 figures, 271 unit tests, and atomic SHA256 manifest covering 419+ artifact files.
