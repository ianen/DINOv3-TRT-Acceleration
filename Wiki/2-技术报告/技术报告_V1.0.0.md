# DINOv3 TensorRT 加速技术报告 V1.0.0

日期：2026-04-30

## 摘要

本报告记录 DINOv3 ViT-L/16 LVD-1689M 在 RTX 5080 / TensorRT 10.13.2.6 上的当前正式实验结果。实验主线完成了 PyTorch/HF 权重到 ONNX、TensorRT engine、真实图片 eval、ModelOpt INT8 定位、locked `trtexec` benchmark、C++ runtime smoke/benchmark 与 Python/C++ parity 的闭环。

当前可用低精度候选是 **BF16 prefer TensorRT engine**。正式 FP16 engine 虽然可以 build 和 benchmark，但在正式 HF 权重上四个输出全 NaN，不能作为有效候选。默认 ModelOpt INT8 Q/DQ 即使用 500 张真实图片校准也仍然塌缩；后段 MatMul partial INT8 可避免塌缩，但速度明显慢于 BF16 prefer，因此只保留为敏感性分析证据。

## ONNX 与 TensorRT 导出

HF Transformers 导出路径使用 `HFDinoV3IntermediateLayerWrapper` 读取 hidden states，选择第 4/12/16/20 层并裁剪 register tokens，保持项目主契约 `[B,197,1024]`。导出前冻结 wrapper/model 参数，避免 ONNX exporter 把可训练参数作为 constant tensor 插入。

DINOv3 RoPE 在导出时会出现 TensorRT 不接受的顶层 `If` 风险。本项目对 HF `DINOv3ViTRopePositionEmbedding` 增加 eval-only ONNX export patch，将 `angles.tile(2)` 改为 `torch.cat((angles, angles), dim=-1)`，正式 ONNX 检查结果为顶层 `If=false`。

### FP16 负例

正式 FP16 engine 在正式权重上可构建、可 benchmark，但 Python TensorRT runtime 对 random normal、zeros、ones、uniform 输入均出现四个输出全 NaN。分 block FP32 回退显示，必须大范围 FP32 回退才能恢复有限输出，性能随之接近 FP32。因此 FP16 只能作为 TensorRT/Blackwell 风险负例，不进入有效候选对比。

### INT8 消融

默认 ModelOpt INT8 Q/DQ 路径在 2 张 smoke calibration 和 500 张真实图片 calibration 下均塌缩。ONNX Runtime 直接比较 FP32 ONNX 与 Q/DQ ONNX 已复现失败，因此问题主要位于量化图/校准配置，而不是 TensorRT engine build 独有问题。

定位结论：

- `MatMul-only` 不零范数，但 cosine 不达候选门限。
- `MatMul+Add` 与 `MatMul+LayerNormalization` 直接复现前三层零范数。
- `MatMul+Mul` 不零范数，但 cosine 明显劣化。
- 后段 MatMul 白名单可以避免塌缩，但量化范围越小，正确性越高、速度收益越低。

INT8 late-layer 消融摘要：

| variant | ONNX 32-image `feat_layer_20` cosine mean | TRT 1000-image `feat_layer_20` cosine mean | locked trtexec speedup vs FP32 |
|---|---:|---:|---:|
| layers16-19 | 0.988997 | 0.989177 | 1.18x / 1.22x / 1.22x |
| layers17-19 | 0.990877 | 未构建 TRT | 未测 |
| layers18-19 | 0.993062 | 未构建 TRT | 未测 |
| layer19 | 0.995601 | 0.995659 | 1.05x / 1.07x / 1.06x |
| layer19_attention | 0.998970 | 0.998994 | 1.04x / 1.05x / 1.04x |
| layer19_mlp | 0.996596 | 未构建 TRT | 未测 |

`layer19` 与 `layer19_attention` 正确性可回到 0.99 以上，但速度只比 FP32 快约 1.04x-1.07x，且明显慢于 BF16 prefer。

### FP8 PTQ（V1.1 stretch goal · 实测 negative result）

V1.0.1 §1.3 V1.1 stretch goals 列出 "FP8 PTQ via TensorRT Model Optimizer (Blackwell 5th-gen Tensor Core)"。本期已开启该实验：

- **量化与构建**：`scripts/quantize_onnx_modelopt.py --quantize-mode fp8` 复用 imagenette `calib_500` 生成 `dinov3_vitl16_4out.fp8.modelopt.imagenette_calib500.onnx`（1.0 GB external data，440 QuantizeLinear/440 DequantizeLinear，opset 19，顶层 `If=false`）；`trtexec --fp8` 构建 engine size 261 MB（FP32 的 1/4，比 BF16 prefer 514 MB / partial INT8 866 MB 还小）。
- **locked trtexec speedup vs FP32**（GPU compute median，batch 1/8/32）：

| reference | candidate | b=1 | b=8 | b=32 |
|---|---|---:|---:|---:|
| FP32 | FP8 ModelOpt | **2.93×** | **4.70×** | **5.05×** |
| BF16 prefer | FP8 ModelOpt | 1.20× | 1.67× | 1.55× |

  这是当前所有候选中**速度最高**的：FP8 batch 32 比 FP32 快 5.05×，比 BF16 prefer 还快 1.55×。

- **正确性**（Imagenette1000 真实图片，FP32 baseline）：

| output | cosine_mean | cosine_min | max abs error | RMSE |
|---|---:|---:|---:|---:|
| `feat_layer_4` | 0.40932 | 0.114245 | 315.5 | 0.9247 |
| `feat_layer_12` | 0.34118 | 0.232097 | 311.1 | 2.3481 |
| `feat_layer_16` | 0.16397 | 0.107876 | 320.8 | 4.6911 |
| `feat_layer_20` | 0.20126 | 0.175263 | 459.1 | 11.5955 |

- **结论**：FP8 默认 PTQ 与 INT8 默认 PTQ **行为一致**——速度收益最大，但精度严重塌缩（cosine_mean 0.16–0.41，远低于 BF16 prefer 的 0.999+）。这强化了既有论证：**Blackwell sm_120 + TensorRT 10.13 + DINOv3 ViT-L/16 + 默认 ModelOpt PTQ** 的组合下，无论 INT8 还是 FP8，默认配置都不能直接产出可部署候选；要恢复正确性必须做节点级 sensitivity analysis。

**FP8 partial layer19**（与 INT8 partial layer19 同口径，imagenette 64 张校准）：

| metric | feat_layer_4 | feat_layer_12 | feat_layer_16 | feat_layer_20 |
|---|---:|---:|---:|---:|
| cos_mean | 1.000000 | 1.000000 | 1.000000 | **0.999410** |
| cos_min | 1.000000 | 1.000000 | 1.000000 | 0.999360 |
| max_abs | 0.0002 | 0.0005 | 0.0021 | 8.21 |

locked trtexec speedup vs FP32 batch 1/8/32：`1.109× / 1.039× / 1.044×`；vs BF16 prefer：`0.45× / 0.37× / 0.32×`。

跟 INT8 partial layer19 对比：

| variant | feat_layer_20 cos_mean | trtexec speedup vs FP32 (b8) |
|---|---:|---:|
| FP8 layer19 | **0.999410** | 1.04× |
| INT8 layer19 | 0.995659 | 1.07× |
| INT8 layer19_attention | 0.998994 | 1.05× |

FP8 partial 在精度上比 INT8 partial 略好（0.99941 vs 0.99566）但速度同样无收益。完整结论：**FP8 与 INT8 在 ViT-L 上的节点级 sensitivity 行为完全对称**，"量化范围越小 → cosine 越高 → speed 越低" 的 trade-off 跨 INT8/FP8 通用。

### SmoothQuant（V1.1 stretch goal · 部分突破默认 PTQ 塌缩）

V1.0.1 §1.3 V1.1 stretch goals 第 2 项 "SmoothQuant ViT-L PTQ"。本期已开启该实验：

- **路径选择**：ModelOpt 0.43 的 SmoothQuant 仅在 PyTorch 路径（`modelopt.torch.quantization` + `INT8_SMOOTHQUANT_CFG`），ONNX PTQ 入口不支持。新增 `scripts/quantize_torch_modelopt_smoothquant.py` 走完整 PyTorch 流程：HF DINOv3 → `make_hf_export_module` 包成 nn.Module → `mtq.quantize(wrapper, INT8_SMOOTHQUANT_CFG, forward_loop=...)` 校准 → `torch.onnx.export` 输出 Q/DQ ONNX → trtexec `--int8` build engine。
- **量化范围**：SmoothQuant 默认只量化 Linear/MatMul 的 weight + activation（242 QuantizeLinear / 242 DequantizeLinear，对比默认 INT8 ONNX 的 442），**主动跳过 LayerNorm/Add**——这正与项目 INT8 sensitivity sweep 中定位的"Add 与 LayerNorm 的 Q/DQ 注入是塌缩触发项"一致。
- **alpha=0.5（SmoothQuant 论文默认）+ 500 张 imagenette 校准** 结果：

| output | cosine_mean | cosine_min | max_abs | RMSE |
|---|---:|---:|---:|---:|
| `feat_layer_4` | **0.985** | 0.981 | 13.76 | 0.141 |
| `feat_layer_12` | 0.953 | 0.945 | 206.86 | 0.681 |
| `feat_layer_16` | 0.919 | 0.904 | 210.13 | 1.549 |
| `feat_layer_20` | **0.919** | 0.894 | 216.79 | 4.029 |

- **locked trtexec speedup**（GPU compute median，batch 1/8/32）：

| reference | b=1 | b=8 | b=32 |
|---|---:|---:|---:|
| FP32 | **2.20×** | **3.50×** | **3.62×** |
| BF16 prefer | 0.90× | **1.24×** | **1.12×** |

  speed 在 batch ≥ 8 时**已超越 BF16 prefer**，且 batch 全档均超过 G2 速度阈值 ≥ 2.2×。

- **结论**：SmoothQuant **部分达成 G2**——速度阈值（≥ 2.2×）已稳定达标，但 cos_min 0.894 仍未达 G2 cosine 阈值（≥ 0.99）。

#### Alpha sweep（500 张 imagenette 校准，分别 alpha = 0.5 / 0.7 / 0.8）

| alpha | feat_layer_4 cos_mean | feat_layer_12 cos_mean | feat_layer_16 cos_mean | feat_layer_20 cos_mean | feat_layer_20 cos_min | speedup vs FP32 (b=8) |
|---:|---:|---:|---:|---:|---:|---:|
| 0.5 | 0.985 | 0.953 | 0.919 | 0.919 | 0.894 | 3.50× |
| 0.7 | 0.992 | 0.993 | 0.981 | 0.978 | 0.963 | 3.49× |
| **0.8** | **0.993** | **0.994** | **0.985** | **0.982** | **0.968** | **3.48×** |

观察：
1. **Alpha 增大显著提升 cosine**：0.5→0.7 让 feat_layer_20 cos_mean 从 0.919 → 0.978（+0.06），cos_min 从 0.894 → 0.963（+0.07）。
2. **Alpha 边际收益递减**：0.7→0.8 仅再 +0.005 (cos_mean) / +0.005 (cos_min)。
3. **速度几乎不变**：alpha 调节只移动 smoothing scale 系数，不改变 Q/DQ 数量与 kernel 选择，所以 speedup 在 3.48-3.50× 区间稳定。
4. **alpha=0.8 是当前最佳 SmoothQuant 配置**：最深层 cos_mean 0.982 / cos_min 0.968；浅层 feat_layer_4 cos_mean 0.993 / cos_min 0.990（cos_min 已达 G2 阈值的浅层！）。

但 cos_min 0.968 仍 < G2 0.99 阈值（缺口 0.022）。alpha 进一步升至 0.9 / 1.0 预期收益已极小。要完整达成 G2，下一步应：
- 敏感层 mixed-precision 回退：feat_layer_16/20（精度差最多的两层）保持 BF16，浅层 INT8 SmoothQuant
- 或更精细的 per-layer / per-head alpha 调节
- 或 SmoothQuant + AWQ 等其它后续算法组合

这是项目首个**真正部分恢复 ViT-L INT8 PTQ 可行性**的方向：相比默认 INT8（塌缩）和默认 FP8（cos 0.20），SmoothQuant 把最深层 cos 从 0 拉到 0.98（feat_layer_20 alpha=0.8）/ 0.99（feat_layer_4 alpha=0.8）。

#### Mixed-precision skip 16-19（V1.1 stretch follow-up · negative-ish result）

承 alpha sweep 提出的"敏感层 mixed-precision 回退"思路，新增 `--skip-blocks 16-19` 让 ModelOpt 在 calibrate 时把 layer 16/17/18/19 的 input_quantizer + weight_quantizer 全部 disable（wildcard `*model.layer.{16..19}.*` -> `enable: False`）。alpha=0.8 / imagenette 500 张校准后实测：

| metric | α=0.8（full SmoothQuant） | **α=0.8 + skip 16-19** |
|---|---:|---:|
| feat_layer_4 cos_mean | 0.993 | 0.993 |
| feat_layer_12 cos_mean | 0.994 | 0.994 |
| feat_layer_16 cos_mean | 0.985 | 0.985 |
| feat_layer_20 cos_mean | 0.982 | **0.984** |
| feat_layer_20 cos_min | 0.968 | **0.971** |
| max_abs_error feat_layer_20 | — | 169.3 |
| trtexec speedup vs FP32 (b=8) | 3.48× | **2.41×** |
| trtexec speedup vs BF16 prefer (b=8) | 1.24× | 0.86× |

观察：
1. **精度提升仅微小**：cos_mean +0.002，cos_min +0.003，**远未跨过 G2 cos_min ≥ 0.99 阈值**。
2. **速度大幅下降 30%**：3.48× → 2.41×（仍 ≥ G2 速度阈值 2.2× 但已临界）；vs BF16 prefer batch ≥ 8 不再有优势。
3. **机制解读**：SmoothQuant 在 calibrate 阶段已经把 smoothing scale **写入了所有 Linear 的 weight**（包括 layer 16-19）。disable 这些层的 quantizer 只是不再插入 Q/DQ 节点，但 TRT INT8 build 时 implicit fallback 让那些层的 weight 经历 dequantize → fp32 compute → quantize 的额外往返，反而拖慢。
4. **结论**：在 ModelOpt SmoothQuant + ONNX export + TRT 路径下，**简单的 disable 子集 quantizer 不是有效的 mixed-precision 配方**。要让 layer 16-19 真正以 BF16 执行需要更深的工具链改造（或者 trtexec `--layerPrecisions` per-layer override），不是本期工程范围。

最终 INT8 路径结论：alpha=0.8 full SmoothQuant 仍是当前最佳 INT8 候选（cos_mean 0.982 / speedup 3.48×），mixed-precision skip 16-19 是有效的 V1.1 stretch 探索但 negative-ish。**BF16 prefer 仍是唯一完整在 G2 ideal region 的候选**。

- **可视化**：`benchmark_bf16_vs_int8_tradeoff.svg` 已扩到 **7 个点**，新加 INT8 SmoothQuant α=0.5（cos 0.919, speedup 3.50×）。该点是当前所有量化候选中**最接近 G2 ideal region** 的：在阈值线 cos=0.99 的左侧但接近，在 speedup=2.2× 的上方且远高于 INT8 partial。

BF16 prefer 仍是唯一**完整**进入 G2 ideal region 的候选，但 SmoothQuant 是**研究方向最有突破潜力**的 INT8 路径。FP8 SmoothQuant、4 层组合 ablation 是后续工作。
- **可视化**：`benchmark_bf16_vs_int8_tradeoff.svg` 已更新为 5 点散点图（BF16 prefer 在 ideal region；FP8 ModelOpt 在最右下角"速度最高但 cosine 最差"；3 种 INT8 partial 在 cosine 阈值附近的 trade-off curve 上），visually 明确"默认 PTQ 都不在 G2 ideal region"。

tradeoff 散点图当前包含 5 个候选点：

- BF16 prefer：cos `0.999127` × speedup `2.81×`，是唯一进入 G2 ideal region (cos ≥ 0.99 ∧ speedup ≥ 2.2×) 的候选。
- FP8 ModelOpt（默认）：cos `0.20126` × speedup `4.70×`，**速度最高但 cosine 严重塌缩**（与默认 INT8 同模式）。
- INT8 layers16-19：cos `0.989177` × speedup `1.22×`，cosine 跌破 G2 阈值且速度未达标。
- INT8 layer19：cos `0.995659` × speedup `1.07×`，cosine 达标但速度跌破。
- INT8 layer19_attention：cos `0.998994` × speedup `1.05×`，cosine 接近 BF16 但速度几乎无收益。

5 个点连起来呈现完整 trade-off：FP8 默认 PTQ 在最右下（最快但塌缩）、INT8 partial 在中下区（量化范围越小 → cosine 越高 → speed 越低）、BF16 prefer 在右上 ideal region 一枝独秀，是当前唯一可作为部署候选的 verdict。

## 4 层组合 ablation（V1.1 stretch goal #3）

为给项目当前 DPT-style 4 层选择 `[4, 12, 16, 20]`（1-based）提供研究证据，新增 `scripts/run_layer_ablation_pytorch.py`：单次 HF DINOv3 forward 以 `output_hidden_states=True` 抓取全部 24 个 transformer block 的隐状态，对三组候选并行计算 inter-output cosine 与 per-output L2 magnitude（Imagenette eval 1000 张, image_size 224, batch 8）。

候选定义（0-based block indices → 1-based feat_layer 命名）：

- **project**：`(3, 11, 15, 19)` → `feat_layer_{4,12,16,20}` — 项目当前主路径，偏前-中段。
- **dpt**：`(4, 10, 16, 22)` → `feat_layer_{5,11,17,23}` — DPT 论文（Ranftl et al., 2021）推荐的 ViT-L 均匀采样。
- **late**：`(5, 11, 17, 23)` → `feat_layer_{6,12,18,24}` — 重尾变体，最深 hook 推到最后一个 block。

**实验结果**（`Code/Artifacts/reports/layer_ablation_pytorch_eval1000_r224.{json,md}`）：

| candidate | layers (1-based) | mean cos | min cos | max cos | magnitude (max/min) |
|---|---|---:|---:|---:|---:|
| project | 4/12/16/20 | **0.3828** | 0.1059 | 0.6506 | **12.6×** |
| dpt | 5/11/17/23 | **0.2990** | 0.0717 | 0.5466 | 31.9× |
| late | 6/12/18/24 | 0.3395 | 0.1438 | 0.6023 | **84.0×** |

Per-output magnitude 对照（mean L2 over `[B, T*C]`）：

- project：362 / 972 / 1753 / 4560
- dpt：398 / 833 / 2137 / 12704
- late：460 / 972 / 2692 / 38652

**结论**：

1. **diversity ranking**（mean inter-output cosine 升序）：dpt < late < project。DPT 的均匀采样确实在 4 输出之间取得了最大特征异质性（mean cos 0.299 vs project 0.383，约 22% 更分散）。
2. **magnitude balance**：project 的 `[4,12,16,20]` 给出最紧的 magnitude 分布（max/min ≈ 12.6×），late 的 `[6,12,18,24]` 把最后一个 block 拉到 38k，比第一层高 ~84×，对 multi-scale fusion 极不友好（最深 hook 会主导融合）。
3. **trade-off**：project 当前选择是"diversity 与 magnitude 均衡"的折中；DPT 选择 diversity 更强但需要 per-scale 归一化才能稳定融合；late 选择两端都不利。
4. **研究证据**：本次 ablation 验证了项目"早一档于 DPT"的层选择不是简单照搬，而是为了避免 ViT-L 最后几个 block 的 magnitude 爆炸——这个特性在没有专门的下游 DPT 头训练前提下，是更稳健的工程默认。

ablation 仅在 PyTorch 端跑（不重导 ONNX、不重 build engine），不影响主路径产物，但为 V1.1 的"层选择"决策提供了可复现的数据支持。

可视化产物：`Code/Artifacts/reports/figures/layer_ablation_diversity_vs_balance.svg` — 三候选在 X 轴 mean inter-output cosine（越低越好）与 Y 轴 max/min magnitude ratio（log10，越低越好）平面上的散点图。lower-left 为 sweet spot；project（蓝色）位于 magnitude 平衡侧、dpt（绿色）位于 diversity 侧、late（红色）位于 magnitude 不平衡侧。

## 结论

1. BF16 prefer 是当前唯一同时满足有限输出、真实图片稳定性、Python/C++ runtime 可用性和明确加速收益的候选。
2. FP16 在正式权重上是失败负例，报告中应单独作为 TensorRT/Blackwell 风险分析。
3. INT8 默认 Q/DQ 主路径未达成；partial INT8 可作为敏感性分析，但不应作为当前部署候选。
4. P5 benchmark 已形成机器可读 matrix；224 分辨率主线完整，336 分辨率有 locked `trtexec` 与 C++ runtime 的 BF16/FP32 batch `1/4/8`，518 分辨率有 locked `trtexec` 与 C++ runtime 的 BF16/FP32 batch `1/2/4`（profile max=4）以及 batch `8`（独立 b8 engine, profile max=8），G3 跨语言一致性同步覆盖 224/336/518 三档。
5. 复现与许可交付已补齐到当前可交付范围；完整 ImageNet val 仍是外部数据 blocker，授权可用后应复用现有 exporter、manifest、eval 与 summary/matrix 入口重跑。
