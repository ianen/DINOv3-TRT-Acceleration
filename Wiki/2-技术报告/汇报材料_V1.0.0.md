## 1. 一段话总结

DINOv3 ViT-L/16 在 RTX 5080 + TensorRT 10.13 上完成了 PyTorch/HF → ONNX → TensorRT engine → 真实图片 eval → INT8 量化定位 → locked `trtexec` benchmark → C++ runtime → Python/C++ 跨语言一致性的完整闭环。**当前唯一的有效低精度部署候选是 BF16 prefer**：相对 FP32，trtexec GPU compute median latency speedup 在 224 batch `1/4/8/16/32` 为 **2.45× / 2.55× / 2.81× / 3.08× / 3.25×**，336 batch `1/4/8` 为 **2.80× / 2.96× / 3.25×**，518 batch `1/2/4/8` 为 **3.12× / 3.50× / 3.76× / 3.86×**；4 输出在 1000 张真实图片上 cosine mean ≥ **0.9991**（**三档分辨率 224/336/518 均覆盖**），无 NaN/Inf/零范数，最低 cosine ≥ 0.9984。FP16 在正式权重上输出全 NaN，是负例；默认 ModelOpt INT8 Q/DQ 即使用 500 张真实图片校准也塌缩，节点级 partial INT8 能恢复正确性但速度慢于 BF16 prefer，因此 INT8 降级为敏感性分析。Python TensorRT runtime 与 C++ TensorRT runtime 在 224 / 336 / 518 三档分辨率上 batch=1 全部 **bit-identical**（`max_abs_error=0`、`cosine=1.0`）。

## 2. 关键决策一览

| 决策 | 选项 | 理由 | 出处 |
|---|---|---|---|
| 主候选低精度 | **BF16 prefer**（不是 FP16，也不是 INT8） | FP16 NaN，INT8 默认塌缩；BF16 prefer 是唯一同时满足正确性、速度与跨语言可执行性的候选 | §3、§4 |
| 输出 token 数 | 197（CLS + 196 patches，**裁剪 4 register tokens**） | 与项目计划 V1.0.1 ADR-001 一致；HF `get_intermediate_layers` 默认行为 | §5 |
| RoPE 处理 | HF `DINOv3ViTRopePositionEmbedding` 源码改造（`angles.tile(2)` → `torch.cat`） | 消除 TensorRT 不接受的顶层 `If`（DINOv3 比 DINOv2 多出的坑） | ADR-007 |
| 多分辨率 | **静态分辨率 + 动态 batch**，每个分辨率独立 ONNX/engine | 197/442/1025 token 数依赖固定输入分辨率 | ADR-009 |
| INT8 主路径 | NVIDIA TensorRT Model Optimizer Q/DQ 显式量化 | `IInt8EntropyCalibrator2` 自 TRT 10.1 deprecated；ModelOpt 是当前官方推荐路径 | ADR-003 |
| 性能口径 | locked GPU clock 2752 MHz + `trtexec --useSpinWait` | 减少 Windows WDDM 100ms 抖动对 p50 的影响 | §3、R10 |
| 数据集口径 | **Imagenette2-320 val** 1000 eval / 500 calib 互斥 | 完整 ImageNet 1k val 仍 HF 403 gated；Imagenette 是公开 ImageNet-style 子集，可复现 | §6 |
| C++ runtime 范围 | RAII 封装 + 持久化 device buffer + `enqueueV3` | 与 Python runtime 共享同一 engine binary；FP32 阈值 MaxAbsErr ≤ 1e-5 | §5 |

| **INT8 SmoothQuant α=0.8 + trtexec --layerPrecisions=l16-19:fp32**（mixed-precision via TRT per-layer override） | **0.9822 (cos_min 0.9683)** | **2.23× / 3.43× / —** | **0.91× / 1.22× / —** |
| **INT8 SmoothQuant α=0.8 ONNX-stripped l16-19**（V1.2 真正的 ONNX graph rewrite mixed-precision） | **0.9842 (cos_min 0.9705)** | **1.72× / 2.39× / —** | **0.70× / 0.85× / —** |

> **结论**：缩小到 `layer19_attention` 后 INT8 正确性可恢复，但加速收益几乎消失，且明显慢于 BF16 prefer。Add 与 LayerNormalization 的 Q/DQ 注入是已确认的塌缩触发项。
>
> **FP8 PTQ（V1.1 stretch）**：默认 FP8 ModelOpt PTQ 在 imagenette 500 张校准下取得**当前所有候选最高速度**（vs FP32 5.05×、vs BF16 prefer 1.55× @ batch 32），但 1000 张真实图片 cosine_mean 0.20（feat_layer_20），与默认 INT8 同模式塌缩。**FP8 partial layer19**（节点级白名单，imagenette 64 张校准）已验证：feat_layer_20 cos_mean 0.99941（精度比 INT8 partial layer19 的 0.99566 还要高），但 trtexec batch 1/8/32 speedup 仅 `1.11× / 1.04× / 1.04×`，与 INT8 partial layer19 同级别（速度收益消失）。FP8 与 INT8 在 ViT-L 节点级 sensitivity 上行为完全对称。
>
> **INT8 SmoothQuant（V1.1 stretch · 部分突破默认 PTQ 塌缩，含完整 alpha sweep）**：用 ModelOpt PyTorch 路径走 `INT8_SMOOTHQUANT_CFG` 后导出 Q/DQ ONNX。SmoothQuant **默认主动跳过 LayerNorm/Add**（242 vs 默认 INT8 442 QDQ），与项目之前定位的塌缩触发项一致。imagenette 500 张校准 alpha sweep：
>
> - alpha=0.5：feat_layer_20 cos_mean 0.919 / cos_min 0.894
> - alpha=0.7：cos_mean **0.978** / cos_min **0.963**（+0.06 cos_mean，主要跨越点）
> - **alpha=0.8 最佳**：cos_mean **0.982** / cos_min **0.968**；浅层 feat_layer_4 cos_min **0.990**（cos_min 已达 G2 阈值的浅层！）
>
> 三档 speedup 几乎不变（alpha 不改变 Q/DQ 数量与 kernel 选择）：vs FP32 batch 1/8/32 ≈ `2.20× / 3.50× / 3.62×`，**全档 ≥ G2 速度阈值 2.2×**；vs BF16 prefer batch ≥ 8 时仍超过（`1.24× / 1.12×`）。**最佳 alpha=0.8 部分达成 G2**（速度达标，cos_min 0.968 仍 < 0.99 阈值，缺口 0.022）。
>
> **Mixed-precision 回退（α=0.8 + skip 16-19）**：用 ModelOpt `disable_quantizer` wildcard 让最后 4 个 transformer block 不被量化。实测 cos_mean 仅 +0.002（0.982 → 0.984）、cos_min +0.003（0.968 → 0.971，**未跨 0.99 阈值**），但 speedup batch 8 从 3.48× 跌到 2.41×（-30%）。机制：SmoothQuant 在 calibrate 阶段已把 smoothing scale 写入所有 Linear weight，disable 子集 quantizer 只让那些层不再插入 Q/DQ，但 TRT INT8 build 仍触发额外 dequantize→fp32→quantize 往返，反而拖慢。**结论：简单的 disable 子集 quantizer 不是有效配方**。
>
> **Mixed-precision 第二次尝试（α=0.8 + trtexec --layerPrecisions=l16-19:fp32）**（V1.1 follow-up · 第十八轮）：构造工程基础设施 `Code/scripts/build_layer_precisions_arg.py` + `Code/scripts/build_mixed_precision_engine_windows.py`，把 100 个 layer 16-19 compute-heavy 节点（32 MatMul + 56 Add + 8 LayerNorm + 4 Softmax）写成 trtexec `--layerPrecisions=node:fp32` argument（3923 chars），用 Python subprocess argv 绕开 cmd.exe 32 KB 限制。**第一次 BF16 override 失败**：TRT 10.13 Myelin Fill 算子不支持 BF16 类型（仅 Int32/Float/Half/Int64），在 RoPE Constant 处 build assertion failure。**第二次 FP32 override 成功 build**：41s 后产出 270 MB engine，trtexec batch 1/8 GPU compute median `3.15 ms / 8.25 ms`（speedup vs FP32 `2.23× / 3.43×`），与 full SmoothQuant α=0.8（3.23/8.15）几乎相同。**Cosine eval (1000 张)**：feat_layer_20 cos_mean **0.9822** / cos_min **0.9683** — **与 full SmoothQuant α=0.8 (0.982/0.968) 完全相同**。机制：Q/DQ 节点位于每个 transformer block 的输入/输出边界，是 TRT 真正的精度边界；`--layerPrecisions` 控制 kernel 内部计算精度（INT8 vs FP32），但 boundary tensor 仍按 Q/DQ scale 量化到 INT8，下游收到的还是 INT8 离散值。**结论：trtexec `--layerPrecisions` 在 explicit Q/DQ ONNX 上对 cosine 是 no-op**。要做真正的 mixed-precision 必须用 onnx-graphsurgeon 在 ONNX 层删除 layer 16-19 周围的 Q/DQ 节点（V1.2 范围）。
>
> **最终 INT8 路径结论**：alpha=0.8 full SmoothQuant 是当前最佳 INT8 候选（cos_mean 0.982 / speedup 3.48×），仍 0.022 cos_min gap 到 G2。BF16 prefer 仍是唯一**完整**在 G2 ideal region 的候选。
>
> **V1.0.0 + V1.1 + V1.2 mixed-precision 三层闭合证据链**（第十四 / 十八 / 二十五轮联合）：
>
> 1. ModelOpt PyTorch 层 `disable_quantizer` skip 16-19（V1.1 第十四轮）：cos_min 0.971，speedup 2.41× — negative-ish。
> 2. trtexec `--layerPrecisions=...:fp32` per-layer override（V1.1 第十八轮）：cos_min 0.9683，speedup 3.43×（≈full） — completely no-op（Q/DQ boundary 才是真精度边界）。
> 3. **V1.2 ADR-010 ONNX graph-level Q/DQ stripping**（V1.2 第二十五轮）：cos_min 0.9705，speedup 2.39× — **与 V1.1 #1 几乎完全相同**（差 0.0005 / 0.02×），DEFINITIVELY 闭合 V1.2 graph rewrite 假设。
>
> **三条路径联合证据指向同一 root cause**：layer 16-19 内部 fp 计算无法 recover blocks 0-15 累积的 INT8 量化噪声。**真正的精度瓶颈在前段，不是末段 mixed-precision 的工具链选择**。要跨过 G2 cos_min 0.99 必须从前段开始减少量化（V1.3 / 论文方向：QAT 量化感知 fine-tuning），不在 V1.0.0 + V1.1 + V1.2 工程范围。
>
> 可视化：`figures/benchmark_bf16_vs_int8_tradeoff.svg`（cosine vs latency speedup 散点图，**12 个点**含 BF16 prefer + 3 INT8 partial + FP8 default + FP8 partial layer19 + 3 SmoothQuant alpha + skip 16-19 + trtexec mixed l16-19:fp32 + V1.2 ONNX-stripped l16-19，标注 G2 阈值线 cos = 0.99 / speedup = 2.2× 与 ideal region 阴影）。BF16 prefer 是唯一进入 ideal region 的候选；三种 INT8 partial 都在阈值左下方的 trade-off curve 上，"量化范围越小 → cosine 越高 → speed 越低"的关系一目了然；V1.2 ONNX-stripped 与 V1.1 ModelOpt skip 16-19 在散点图上几乎完全重合，视觉化呈现"两条 mixed-precision 路径等价"的实证结论。

### 4.2 FP16 负例

正式 FP16 engine 在正式权重下可 build / benchmark，但 Python TensorRT runtime 对 random-normal / zeros / ones / uniform 四种输入均输出 **100% NaN**。分 block FP32 诊断显示，需要把所有 24 层都强制 FP32 才能恢复有限输出（性能基本退化为 FP32）。因此 FP16 在 Blackwell + TRT 10.13 + DINOv3 这一组合下是结构性失败，应作为风险负例报告而不是候选。

## 7.5 4 层组合 ablation（V1.1 stretch goal #3）

为给项目当前 DPT-style 4 层选择 `[4,12,16,20]` 提供研究证据，新增 `scripts/run_layer_ablation_pytorch.py`：单次 HF DINOv3 forward 抓取 24 个 transformer block 的隐状态，对三组候选并行计算 inter-output cosine 与 per-output L2 magnitude（Imagenette eval 1000 张, 224, batch 8）。

| candidate | layers (1-based) | mean cos | min cos | max/min magnitude |
|---|---|---:|---:|---:|
| project | 4/12/16/20 | **0.3828** | 0.1059 | **12.6×** |
| dpt | 5/11/17/23 | **0.2990** | 0.0717 | 31.9× |
| late | 6/12/18/24 | 0.3395 | 0.1438 | 84.0× |

**结论**：DPT 均匀采样的 inter-output diversity 最高（mean cos 0.299），但 magnitude max/min 已达 32×；project 的"早一档于 DPT"选择把 magnitude 不平衡度压到 12.6× 的同时保持 mean cos 0.383，是 diversity-magnitude 的折中——不是简单照搬 DPT 论文，而是为了避免 ViT-L 最后几个 block 的 magnitude 爆炸（late 变体 L24 的 magnitude ~38k，比 L6 高 84×，对 multi-scale fusion 极不友好）。该 ablation 仅在 PyTorch 端跑（不重导 ONNX、不重 build engine），但通过 `contracts.DINO_VITL16_LAYER_ABLATION_CANDIDATES` + `make_dinov3_vitl16_contract(layer_indices=...)` 已把"切换 4 输出绑定"做成参数化操作，下游 DPT 头训练时可一行 override。

## 10. 一句话答辩

> 我们用 BF16 prefer 在 RTX 5080 上把 DINOv3 ViT-L/16 的 4 输出多尺度推理在 224/336/518 三档分辨率上稳定加速了 2.45×–3.86×（trtexec GPU median）和 2.27×–3.40×（C++ end-to-end），4 输出在每档分辨率 1000 张真实图片上 cosine ≥ 0.998（最深层 `feat_layer_20` 在 r518 上 cosine_mean 0.999721 反而高于 r224，因为更多 patch token 把误差稀释），Python 与 C++ runtime 跨三档分辨率均 bit-identical；FP16 是 NaN 负例，INT8 是塌缩-恢复-but-no-speed 的敏感性分析，FP8（V1.1 stretch）默认 PTQ 速度最高（5.05× vs FP32 / 1.55× vs BF16）但 cosine 同样塌缩到 0.20，强化"默认 ModelOpt PTQ 不能直接产出 ViT-L 候选"的论证；完整 ImageNet 是外部数据 blocker，授权一到位即可一键替换重跑。
