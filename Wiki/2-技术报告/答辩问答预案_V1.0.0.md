## Q1 · 为什么主候选是 BF16 prefer，不是 FP16？

**简短答案**：FP16 在正式 DINOv3 ViT-L 权重上**输出全 NaN**（已确认负例），是 Blackwell sm_120 + TRT 10.13 + DINOv3 RoPE 的工程性 incompatibility。BF16 prefer 在所有真实图片上输出有限、最低 cosine ≥ 0.998，是唯一同时满足 G1+G3 的低精度候选。

**详细论据**：
- FP16 GPU compute 速度本身没问题（trtexec 跑得快），但 1000 张真实图片 forward 后 4 个 feat_layer 全 NaN。
- BF16 prefer 在 224/336/518 三档分辨率 1000 张 Imagenette eval 中：feat_layer_20 cos_min `0.998749 / 0.998604 / 0.999171`（r518 反而最高，因为更多 patch token 把误差稀释）。
- BF16 prefer 速度（locked 2752 MHz + spin-wait）：
  - r224 b1/b8/b32：vs FP32 `2.45× / 2.81× / 3.25×`
  - r518 b1/b8：vs FP32 `3.12× / 3.86×`（顶点）

## Q2 · 为什么 INT8 候选的 cos_min 都 < 0.99？

**简短答案**：**前段（blocks 0-15）累积的 INT8 量化噪声是 root cause**。V1.0.0+V1.1+V1.2 三层 mixed-precision 工具链（PyTorch ModelOpt disable / TRT layerPrecisions / ONNX graph strip）都试过——结果**等价 negative**。改 block 16-19 内部精度只能让那 4 个 block 的 inference 更精确，但不能修复输入污染。

**详细论据**（V1.0.0+V1.1+V1.2 联合闭合证据链，详见 ADR-010 § 5.3 + ADR-011 § 2）：

| 路径 | 工具链层 | 实施轮次 | feat_layer_20 cos_min | b8 speedup |
|---|---|---|---:|---:|
| Full SmoothQuant α=0.8 | PyTorch ModelOpt | V1.1 第 12 轮 | 0.968 | 3.48× |
| ModelOpt `disable_quantizer` skip 16-19 | PyTorch | V1.1 第 14 轮 | 0.971 (+0.003) | 2.41× (-30%) |
| trtexec `--layerPrecisions=l16-19:fp32` | TRT command-line | V1.1 第 18 轮 | 0.9683 (≈) | 3.43× (≈) |
| **V1.2 ONNX graph-level Q/DQ stripping** | ONNX library | **V1.2 第 25 轮** | **0.9705** | **2.39×** |

三种工具链产生几乎完全相同的精度/速度结果（cos_min 差 0.003 内、speed 差 0.02× 内），DEFINITIVELY 闭合 V1.0.0+V1.1+V1.2 全部 mixed-precision 路径。

**要跨过 G2 cos_min ≥ 0.99**：必须从前段开始减少量化（QAT 量化感知 fine-tuning，V1.3 方向，ADR-011）。

## Q3 · V1.0/V1.1/V1.2 整体结论是什么？

**简短答案**：BF16 prefer 是 RTX 5080 + TRT 10.13 + DINOv3 ViT-L 上唯一在 G2 ideal region（cos ≥ 0.99 ∧ speedup ≥ 2.2×）的候选。INT8 全部路径作为 sensitivity analysis 已完整 — 默认 PTQ 塌缩、节点级 PTQ 速度收益消失、SmoothQuant 速度达标但精度差 0.022、所有 mixed-precision 路径等价 negative。**V1.0.1 主计划 G1-G5 验收完整闭合**。

**详细论据**：

| Goal | V1.0.1 验收口径 | 状态 | 主候选 |
|---|---|---|---|
| G1 低精度加速 | FP16 ≥ 1.5× 精度无损 | 部分达成（FP16 NaN 替代为 BF16 prefer） | BF16 prefer 三档分辨率 2.45× ~ 3.86× |
| G2 INT8 加速精度 | INT8 ≥ 2.2× ∧ cosine ≥ 0.99 | 未达成（降级为敏感性分析） | SmoothQuant α=0.8 cos 0.982/speed 3.48× |
| G3 Python/C++ 一致 | FP32 / 低精度 / INT8 按分档一致 | 已达成 | 224/336/518 batch 1 全 bit-identical |
| G4 benchmark 矩阵 | 3 精度 × batch × 分辨率 + 稳定统计 | 已达成 | 56 行 CSV + 8 SVG |
| G5 可复现 | code + README + license + manifest | 已达成 | atomic SHA256 manifest + 一键 sync 工具 |

## Q4 · 4 层选择 `[4, 12, 16, 20]` 怎么选的？是不是 DPT 论文照搬？

**简短答案**：不是 DPT 简单照搬。第 15 轮 ablation 实验在 1000 张真实图片上对比 3 组候选（project / dpt / late）的 inter-output cosine + per-output magnitude，证明 project `[4,12,16,20]` 是**diversity-magnitude 折中**，避免 ViT-L 末层 magnitude 爆炸。

**详细论据**（M1-progress 第 15 轮，PyTorch hooks 实验）：

| candidate | layers (1-based) | mean cos | max/min magnitude |
|---|---|---:|---:|
| **project** | 4/12/16/20 | **0.383** | **12.6×** ← 最平衡 |
| dpt | 5/11/17/23 | **0.299** ← 最分散 | 31.9× |
| late | 6/12/18/24 | 0.339 | **84.0×** ← 最不平衡 |

DPT 均匀采样 inter-output diversity 最高（mean cos 0.299），但 magnitude max/min 已达 32×。late 变体的 L24 magnitude ~38k 比 L6 高 84×，对 multi-scale fusion 极不友好。Project 当前选择把 magnitude 不平衡度压到 12.6× 同时保持 mean cos 0.383，是不需要 per-scale LayerNorm 即可稳定融合的工程默认。

## Q5 · RTX 5080 速度怎么样？多分辨率扩展？

**简短答案**：BF16 prefer locked + spin-wait benchmark：r224 b8 **2.81×**、r336 b8 **2.96×**、r518 b8 **3.86×**。多分辨率 r518 batch=8 是项目顶点，端到端 C++ runtime b8 r518 速度也 3.40×。

**详细论据**（含 4 张 benchmark SVG）：

| 分辨率 | b1 trtexec | b8 trtexec | b1 cpp | b8 cpp |

| 224 | 2.45× | 2.81× | 2.27× | 2.50× |
| 336 | 2.80× | 2.96× | 2.60× | 2.85× |
| 518 | 3.12× | **3.86×** | 2.83× | **3.40×** |

锁频 2752 MHz + `--useSpinWait` 确保 trtexec GPU compute median 与 trimmed median 双指标稳定，规避 Windows WDDM 100ms 抖动。

## Q6 · Python/C++ 跨语言一致性怎么保证？

**简短答案**：Python TensorRT runtime 与 C++ TRT runtime 用同一份 deterministic sine input、同一 engine binary、同一精度档，4 输出**逐个 bit-identical**（max_abs_error=0、cosine=1.0），224/336/518 三档分辨率 batch=1 全部覆盖。

**详细论据**：

- 工具链：MSVC + Visual Studio 2022 Developer Command Prompt + CUDA runtime + TensorRT 10.13.2.6 C++ libs。
- 测试方法：`dinov3_trt_dump_outputs.exe` 写出 C++ runtime 的 4 个 feat_layer tensor 为 float32 binary，`scripts/compare_cpp_python_parity.py` 用 Python TRT runtime 跑同输入并逐输出比较。
- 覆盖：r224 FP32 / BF16 prefer / partial INT8 / layer19 INT8；r336 FP32 / BF16 prefer；r518 FP32 / BF16 prefer。
- 结果：12 份 `cpp_python_parity_*.json` 全部显示 4 输出 max_abs_error=0、RMSE=0、cosine=1.0。

## Q7 · 完整 ImageNet 数据问题怎么办？

**简短答案**：Hugging Face `ILSVRC/imagenet-1k` 是 gated repo，Windows 端没有可用代理直连超时；当前主路径用 **Imagenette2-320 val（1000 eval / 500 calib 互斥）**作为 Imagenet 替代，一旦 unblock 直接用 `scripts/export_hf_imagenet_parquet_images.py` 一键替换，所有 manifest / eval / summary / matrix 入口可复用，不需重写。

**详细论据**：

- HF gated repo 403：在 RTX 5080 Windows + 多种代理路径下都验证 unable to download；按用户指令"按指令不重试"，不浪费时间在外部 blocker 上。
- 替代方案：Imagenette2-320 val 是 ImageNet 子集（10 类 13K 张），与 DINOv3 LVD-1689M 预训练 distribution 接近。
- 替换方案：`scripts/export_hf_imagenet_parquet_images.py` + `scripts/select_imagenet_subset.py`（设计已验证），unblock 后一条命令重生 manifest，所有下游脚本（calibration / eval / cosine summary / matrix / figures）可机械复用。

## Q8 · INT8 既然 cos_min < 0.99，未来怎么解决？

**简短答案**：V1.3 方向是 **QAT 量化感知 fine-tuning**（ADR-011 完整设计文档）。从 V1.1 SmoothQuant α=0.8 PTQ 初始化出发，用 ImageNet val 50K + ModelOpt QAT mode + 1-5 epoch fine-tune，期望 cos_min ≥ 0.99 同时保留 ≥ 3.0× speedup。**未启动**因为 4 条门槛中至少一条（数据集 unblock）未满足。

**详细论据**（ADR-011 § 8 启动判定门槛）：

| 启动条件 | 当前状态 |
|---|---|
| 数据集 unblock（ImageNet val 50K 或 ≥10K 等价） | ❌ HF gated 仍 blocked |
| 训练资源到位（A100/H100 ≥ 5 GPU-day 或 RTX 5080 ≥ 1 week） | ❌ 当前只有 RTX 5080 |
| 专项时间预算（1-2 个月，含论文写作） | ❌ 本期 V1.0+V1.1+V1.2 已耗 ~6 周 |
| 下游任务 baseline 就位（depth/segmentation FP32 + DPT 头训练 pipeline） | ❌ 未启动 |

不是工程懒惰，是 4 条门槛全未满足。ADR-011 § 6 实施步骤、§ 7 风险与缓解（5 条）、§ 5 期望结果矩阵都已就位，等条件满足直接可启动。
